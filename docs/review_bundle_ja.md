## Phase1 dispute の位置づけ
- Phase1 の dispute は、FairSoil 全体の中心機能ではなく、`Token A / Token B / escrow 付き work agreement` を補助する**例外処理**として置いている。
- 目標は「完全な自動裁判」ではなく、**一方的な踏み倒しや長引かせで弱い側が潰れにくい最小手続**を提供することにある。
- したがって README では dispute を重い制度として説明しすぎず、Phase1 では「軽い review と最終確定の仕組み」として扱う。

## Phase1 dispute の原則
- 裁定結果は wallet size ではなく `evidence / timeline / procedure` を主軸に判断する。
- integrity は補助指標であり、証拠より強い根拠として扱わない。
- defense quota は「争えるようにする」ための補助であって、自動勝利補正ではない。
- requester 側保護は完全対称の dispute ではなく、**限定的な misconduct report** として扱う。
- 裁定ログは `Executed by ... as ...` と evidence 参照を残し、後から検証しやすくする。
- Phase1 の internal review は既定路線だが、**重い案件は将来 external adjudication に逃がせる socket** を維持する。

# FairSoil Specification (JA)

このドキュメントは README_ja.md の詳細仕様です。入口は README_ja.md を参照してください。

## コアメカニズム

### 1. デュアルトークンシステム
- **トークン A（Flow）:** 日常取引のための「減価」する通貨。
  - 認証済みのすべての人に毎日配布。
  - 時間とともに自動的に価値/量が減少（例: 30 日の半減期）し、流通を促進。
  - **受取方式:** 日次で発生し、未受領枠として蓄積できる。受取は日次/週次/月次など任意タイミングで一括請求可能。
  - **未受領枠の減価:** 30日まで減価なし、31日以降は通常の減価率を適用（**各日次発生分は発生日タイムスタンプを保持し、請求時に経過日数に応じて一括減価**。分割請求での減価回避を防ぐ）。
  - **自己制限:** ユーザーが「引き出し頻度/上限」を自分で設定できる。
  - **代理設定:** 期限付き・本人撤回権・監査ログを前提に、信頼できる代理が受取プロファイルを補助できる。
    - **ガードレール:** 有効期間の上限 (例: 90日) と自動失効。
    - **ガードレール:** 代理の変更・解除は本人が即時可能。拒否権は常に本人が保持。
    - **ガードレール:** 代理による設定変更は監査ログに記録され、第三者検証可能。
    - **ガードレール:** 代理の権限は「受取頻度/上限の設定」に限定し、送金先の変更や資産移動は不可。
    - **ガードレール:** 代理権限の付与/更新には本人の明示同意と再確認を必須とする。
- **トークン B（Asset）:** 永続的な価値の保管手段。
  - 高い誠実性の行動や社会的に不可欠な貢献を通じて獲得。
  - トークン A の単純な貯蔵では獲得できない。
  - **ロック仕様（実装準拠）:** Covenant のエスクローで Treasury が `lockB` を実行し、  
    ロック残高は送金不能・流通量から除外される。解除時は `unlockB`。

### 2. 誠実性に基づく評価
- 報酬は「誠実性」（単に「良いことをする」ではなく「不正をしない」）に基づく。
- 「コストの外部化」（リスク/コストを他者に押し付けること）を罰する。
- 安全性および手順基準が満たされたことを検証するために、プロセスの証明を使用。

### 3. Sybil 耐性
- **初期実装:** **World ID** と統合し、各参加者が一意の人間であることを保証。
- **モジュール性:** ID 検証は差し替え可能なモジュールとし、複数IDプロバイダ/ローカルKYC/コミュニティ証明に対応できる設計とする。
- **多層ID（実装方針）:**  
  - Tier1: メール/SNS 等の軽量参加（閲覧・お試し）  
  - Tier2: Passkey（端末生体）で少額の権限を段階開放  
  - Tier3: World ID または ZK‑NFC（公的IDのNFC証明。例: パスポートIC/マイナンバー等）でフル受給  
  - **Nullifier 分離:** UBI/投票/統計で用途別に一意性を分離  
  - **端末バインディング:** 盗難耐性を高め、端末変更時は再認証で再バインド
  - **再認証の条件:** 端末変更・プライマリアドレスの再バインド・高リスク操作（前払い枠の拡張等）のみで要求する（毎回の提示は不要）
  - **ZK‑NFC 検証API（最小仕様）:**  
    - **POST** `ZKNFC_VERIFIER_URL`  
    - **入力(JSON):** `{ "address": "0x..." }`  
    - **返却(JSON):** `{ "verified": true }` または `{ "verified": false, "message": "..." }`  
    - **備考:** 成功時に on‑chain の `setPrimaryAddress(address,true)` を実行する

### 4. 社会的に不可欠なタスクへの動的インセンティブ
- 供給が不足している緊急・高リスク・必須タスク（例: 災害、急患）に対して、報酬を自動的にスケール。

## 仕様 (案)

### 0. 会計ルール草案（A/B/Treasury）
以下は設計の「破綻回避」を目的とした最小会計定義。実装・ガバナンスはこの表に従う。

#### 0-1. トークン種別と役割
- **Token A (Flow):** UBI と日常取引の流通通貨。減価（バーン）がある。
- **Token B (Asset):** 長期価値・貢献の報酬。減価なし（原則）。
- **Soil Treasury:** A/B の支出・回収・再配分を行う金庫。会計上の唯一の「公的支払主体」。

#### 0-2. 会計台帳（最小）
各イベントは以下のどれかに必ず分類される。分類されない支払は許可しない。  
**重要:** 「Burn（消却）」は会計上の収入ではない（入金ではなく供給縮小＝回収手段）。Treasury の支払原資は **(a) Treasury が実際に受け取った収入** または **(b) 明示ルールに基づく新規発行（Seigniorage）** のみで構成する。

**Token A**
- **Mint(A):** UBI 配布、Treasury 支出の原資としての発行（※発行理由をタグ付け）
- **Burn(A):** 減価、罰則、手数料などによる消却（※Burn した量は Treasury 残高に入らない）
- **Transfer(A):** ユーザー間移転
- **Lock(A):** 掟への一時ロック（必要に応じて許可）
- **Unlock(A):** ロック解除

**Token B**
- **Mint(B):** 掟の成果承認、耐久性ロイヤリティなどの貢献報酬
- **Burn(B):** 罰則、資源還流税の一部（必要に応じて許可）
- **Transfer(B):** ユーザー間移転
- **Lock(B):** 掟ステーキング、供託、担保
- **Unlock(B):** ロック解除

**Treasury（最小会計）**
- **収入（Treasury_In）:**  
  - **Fees_to_Treasury:** 手数料（Treasury が受領した分）  
  - **Taxes_to_Treasury:** 還流税（Treasury が受領した分）  
  - **Slashing_to_Treasury:** スラッシュ（没収が Treasury に帰属する分）  
  - **External_In:** 外部からの寄付・助成・ブリッジ等（任意）
  - **実装メモ:** 収入の記録は `recordFee/recordTax/recordSlashing/recordExternalIn` のみを用い、汎用 `recordTreasuryIn` は外部公開しない
- **支出（Treasury_Out_A）:** 公的タスク報酬、UBI補助、Targeted Support、移行期のUBI上乗せ 等
- **支出（Treasury_Out_B）:** 誠実貢献報酬、撤退・廃棄の完遂ボーナス、資源還流報酬 等
  - **支出理由（Reason）:**  
    - A系: `UBI`, `UBI_CLAIM`, `DEFICIT`  
    - B系: `ADVANCE`, `TASK`, `CRYSTAL`
  - **収入理由（Reason）:**  
    - `FEE`, `TAX`, `SLASH`, `EXTERNAL`
  - **記録API対応:**  
    - `recordFee` → `FEE`  
    - `recordTax` → `TAX`  
    - `recordSlashing` → `SLASH`  
    - `recordExternalIn` → `EXTERNAL`
  - **負債理由（Reason）:**  
    - `ADV_LIAB`（前借りの負債化）  
    - `ADV_SETTLE`（前借りの負債解消）  
    - `COV_CREATE`（掟作成での負債化）  
    - `COV_SETTLE`（掟決済での負債解消）
  - **返済メモ:** `settleAdvanceB(from, amount)` は `from` が指定されている場合、`transferFrom` により B を回収して解消する
  - **返済メモ（補足）:** `from = 0x0` の場合は外部回収を行わず、Treasury保有残高の範囲で解消する
  - **運用メモ:** 返済の実行主体は `Treasury`（管理者/ガバナンス）とし、返済ルールはガバナンス決定で更新可能とする
  - **監査メモ:** `advanceBOutstanding + advanceBSettledTotal == Advance(B)発行総量` を満たすこと
  - **インセンティブ案:** 前借り返済者に少額の B ボーナスを付与、あるいは誠実スコアの加点を検討
  - **実装メモ:** reason 定数は `SoilTreasury.REASON_*` としてオンチェーン参照可能
- **発行（Seigniorage）:** ルールに基づく新規発行（例：A の減価に連動する補填ミント 等）  
  - ※Seigniorage は「収入」ではなく「供給操作」として別枠管理する
- **負債（Liabilities）:** 未来のB報酬（前借り分）、未決の掟報酬、救済クレジット 等
- **準備金（Reserves）:** Treasury が実際に保有する A/B 残高（支払能力の上限）
  - **実装メモ:** Liabilities は `adjustLiabilities(deltaA, deltaB)` で増減し、Reserves は `snapshotReserves()` で記録する
  - **運用メモ:** Reserves のスナップショットは「週次/重要イベント後/緊急時」に最低1回実施し、監査に利用する
- **支払判定の目安:** `canPayOutA(amount)` / `canPayOutB(amount)` で準備金/赤字枠の範囲内かを事前判定できる
  - **実装メモ:** UBI/緊急発行/報酬ミントは `canPayOut` によるガードを通過しないと実行できない

> 補足（推奨）: 手数料は「(1) Treasuryが受領 → (2) 必要に応じて一部Burn」という順序で扱う。  
> こうすることで「Burnが収入に見える」誤解を防ぎ、監査ログ上も追跡可能になる。

#### 0-3. 一貫性ルール（破綻回避の最低条件）
- **R1: 会計の一意性**  
  任意の支払は必ず「Mint(A/B) or Treasury支出 or 既存残高移転」に分類される。
- **R2: 原資の明示（Treasury 支払制約）**  
  Treasury からの支出は、必ず **(a) Treasury が実際に保有する準備金（Reserves）**、または  
  **(b) 明示された発行ルール（Seigniorage）** によって裏付ける。  
  **Burn は収入ではない**ため、「Burn したから支払える」という会計は許可しない。
- **R3: 減価とミントの関係**  
  「減価（Burn）に連動してミントを行う」場合は **A の Burn 指標を参照して A を Mint する**のみ許可する。  
  **A の Burn → B の Mint は禁止**。A→B は「結晶化」専用ルール（交換率・上限・手数料）を必須とする。
- **R4: 二重支払の禁止**  
  同一事象を A と B の二重報酬にしない。併給する場合は明示し、上限を定める。
- **R5: ロックと流通量の分離**  
  Lock(B) は流通量に含めない。Unlock 時のみ流通へ復帰する。  
  **実装メモ:** `lockedBalance` / `totalLocked` / `unlockedBalanceOf` が参照可能。
- **R6: 未来債務の上限**  
  Emergency Advance や前払いは「発行総量のX%以内」などの上限を設定する。
- **R7: 支払能力の監査（必須ログ）**  
  Treasury は定期的に `Reserves_A/B`, `Treasury_In`, `Treasury_Out_A/B`, `Seigniorage_A`, `Liabilities` を公開し、  
  支出が R2 を満たすことを誰でも検証可能にする。

#### 0-4. 交換率・上限（A -> B 結晶化）
- **交換率:** 市場価格 + 保守的ディスカウント（例: TWAP * 0.7）
- **上限:** 1ユーザーあたり日次/週次の上限を設定
- **手数料:** A の自然減価損失を下回らない範囲で動的設定

#### 0-5. 制御優先順位（最小の指針）
1. サーキットブレーカー（停止/凍結）
2. APPI連動（A配布量・減価率、外部CPIは補助）
3. Survival Buffer（減価の下限保証）
4. Targeted Support（誠実困窮支援）
5. Emergency Advance / 前払い

#### 0-6. 数式版（簡易・暫定）
**目的:** 実装と検証の共通基準を作るための最小式。詳細は後続仕様で更新。

**一覧テーブル（記号/意味/式）**
| 記号 | 意味 | 式/定義 |
| --- | --- | --- |
| `A_t` | 時点 `t` の A 総供給 | `A_{t+1} = A_t + M_A(t) - B_A(t)` |
| `B_t` | 時点 `t` の B 総供給 | `B_{t+1} = B_t + M_B(t) - B_B(t)` |
| `UBI_base(t)` | 1人あたり日次配布（無条件） | `M_A(t) = UBI_base(t) * N + TS_t + Seigniorage_A(t)` |
| `UBI_total(t)` | 1人あたり日次配布（合計） | `UBI_base(t) + TS_t/N + OtherAdditions(t)`（例: 移行期のUBI上乗せ等） |
| `d_t` | A の日次減価率 | `B_A(t) = d_t * max(0, A_t - SB_t * N) + Fees_A(t)` |
| `SB_t` | Survival Buffer | 個人減価: `a_i(t+1)=a_i(t)-d_t*max(0,a_i(t)-SB_t)` |
| `L_B(t)` | B のロック総量 | `Circulating_B(t) = B_t - L_B(t)` |
| `TS_t` | Targeted Support 追加配布 | `M_A(t)` に加算 |
| `EA_t` | Emergency Advance | `AdvanceCap_B(t)` の範囲で B 前借り |
| `P_AB(t)` | A→B 交換率 | `TWAP_B(t)*(1-haircut)` |
| `DeficitCap_A(t)` | Treasury 赤字上限（A） | `min( B_t の 1%〜5%, Reserves_A(t) )` |
| `AdvanceCap_B(t)` | B 前借り上限 | `min( B_t の 0.5%〜2.0%, Reserves_B(t) )` |
| `APPI_t` | 自律型購買力インデックス | 生存枠バスケットの中央値 |

**記号**
- `A_t, B_t`: 時点 `t` の総供給
- `a_i(t), b_i(t)`: ユーザー `i` の保有量
- `L_B(t)`: Lock(B) 総量
- `C_t`: 外部CPI 指標（補助）
- `UBI_base(t)`: 日次 A 配布量（1人あたり・無条件）
- `UBI_total(t)`: 日次 A 配布量（1人あたり・合計）
- `d_t`: A の日次減価率
- `SB_t`: Survival Buffer（A 残高の減価免除下限）
- `TS_t`: Targeted Support で追加配布される A 量
- `EA_t`: Emergency Advance による B 前借り発行量
- `APPI_t`: 自律型購買力インデックス（内部CPI）
- `Seigniorage_A(t)`: ルールに基づく A の新規発行（Treasury支払のための明示ミント）
- `Treasury_In(t), Treasury_Out(t)`: Treasury の実収入/実支出
- `Reserves(t)`: Treasury が実際に保有する A/B 残高
- `Treasury_Out_A(t), Treasury_Out_B(t)`: Treasury の A/B 支出
- `Reserves_A(t), Reserves_B(t)`: Treasury が実際に保有する A/B 残高
- `DeficitCap_A(t)`: Treasury 赤字上限（A）
- `AdvanceCap_B(t)`: B 前借り上限
- `M_A(t), B_A(t)`: A の Mint/Burn
- `M_B(t), B_B(t)`: B の Mint/Burn

**A の供給と残高**
- `A_{t+1} = A_t + M_A(t) - B_A(t)`
- `M_A(t) = UBI_base(t) * N + TS_t + Seigniorage_A(t)`
- `B_A(t) = d_t * max(0, A_t - SB_t * N) + Fees_A(t)`
- 個人の減価: `a_i(t+1) = a_i(t) - d_t * max(0, a_i(t) - SB_t)`

**B の供給と流通**
- `B_{t+1} = B_t + M_B(t) - B_B(t)`
- `Circulating_B(t) = B_t - L_B(t)`
- `M_B(t) = CovenantRewards(t) + DurabilityRoyalty(t) + ...`
- `B_B(t) = Slashing(t) + ReinvestmentBurn(t)`

**A -> B 結晶化**
- `Mint_B_from_A(t) = min( Cap_i,  A_used / P_AB(t) )`
- `P_AB(t) = TWAP_B(t) * (1 - haircut)`（例: haircut=0.3）
- 結晶化手数料 `fee_AB(t)` を課し、`fee_AB(t) >= d_t` を満たす範囲で設定

**Treasury 収支制約（最小）**
- `Treasury_Out_A(t) <= Treasury_In(t) + Seigniorage_A(t) + Reserves_A(t)`
- `Treasury_Out_B(t) <= Reserves_B(t)`（B の支払は準備金内で実行）
- `DeficitCap_A(t)`: 0-7 の上限定義に従う（`min( B_t の 1%〜5%, Reserves_A(t) )`）。
- `AdvanceCap_B(t)`: 0-7 の上限定義に従う（`min( B_t の 0.5%〜2.0%, Reserves_B(t) )`）。
- **注記:** 「当期支払可能額 = 当期流入 + 当期発行 + 期首準備金」を前提とし、期末準備金は `Reserves_A(t+1) = Reserves_A(t) + Treasury_In(t) + Seigniorage_A(t) - Treasury_Out_A(t)` で更新する。

#### 0-7. 初期値・推奨レンジ（暫定）
実験フェーズ向けの安全レンジ。運用はガバナンスで更新する。

**A（UBI/減価）**
- `UBI_base(t)`: 1人あたり日次 A 配布量（無条件）。初期は「最低生活費の1/30」を目安。
- `d_t`: 日次減価率の初期値 1.0%〜3.0%（半減期 約23〜69日）。  
  週次の変更幅は ±0.5% 以内。
  - **根拠:** 半減期が短すぎると生活安定を崩し、長すぎると循環が鈍るため中庸域を採用。

**Survival Buffer**
- `SB_t`: 「APPIの30日移動平均」で推定。週次更新、変化幅 ±3%/週。
  - **根拠:** 指数はノイズが大きいため移動平均と更新幅制限で過剰反応を抑える。

**Targeted Support**
- 閾値: 「30日平均資産が SB_t の 0.7〜1.0 倍以下」  
- 追加配布: `UBI_base(t)` の 10%〜30% を上限。
  - **根拠:** 救済の厚みはUBIの補助に限定し、過剰な逆インセンティブを避ける。

**Emergency Advance**
- `EA_t`: 1イベントあたり `B_t` の 0.5%〜2.0% を上限。  
- 返済回収比率（天引き）: 日次 UBI の 5%〜30% を範囲に設定。
  - **根拠:** 前借りが常態化しないよう総量上限と返済速度の両方に制約を置く。

**A -> B 結晶化**
- `haircut`: 0.2〜0.4（例: 0.3）  
- `Cap_i`: 1人あたり日次/週次上限を設定（例: 0.5% of B_t / N）
  - **根拠:** ヘアカットは減価回避の収益化を防ぎ、上限は自己取引の経済的魅力を下げる。

**Treasury 収支**
- `DeficitCap_A(t)`: `min( B_t の 1%〜5%, Reserves_A(t) )` を上限。  
  30日移動平均で制御し、急変を抑える。
  - **根拠:** 公的支出の柔軟性を確保しつつ、恒常的赤字と準備金超過を避ける。
 - `AdvanceCap_B(t)`: `min( B_t の 0.5%〜2.0%, Reserves_B(t) )` を上限。  
   30日移動平均で制御し、急変を抑える。
   - **根拠:** 前借りの濫用を防ぎつつ、非常時の柔軟性を確保する。

**APPI（自律型購買力インデックス）**
- 更新頻度: 週次（異常時は日次に切替）
- 除外ルール: 中央値から±3σを外れたデータは除外
- バスケット更新: 月次で品目入替・ウェイト調整

#### 0-8. 会計上の禁止事項（最小）
以下は破綻回避のための「禁止ルール」。例外はガバナンス承認＋タイムロック必須。

- **二重支払の禁止:** 同一事象を A と B の二重報酬にしない（併給時は明示上限）。
- **AのBurn→BのMint禁止:** Aの減価分を直接Bへ変換しない。A→Bは結晶化ルールのみ。
- **Treasuryの隠れ赤字禁止:** Treasury支出は `Reserves` か `Seigniorage` で裏付け、`DeficitCap_A`/`AdvanceCap_B` を超えない。
- **ロック残高の誤認禁止:** Lock(B) を流通量に含めた指標計算を禁止する。
- **自己取引の黙認禁止:** 実質同一主体の循環取引を検知した場合は結晶化無効化/低率適用。
- **スコア不可侵領域の侵害禁止:** UBI、Survival Buffer、異議申立て、再審、退出権はスコアで制限しない。

### 経済・分配
#### 1. ハイブリッド動的報酬と公的プール (インフラと緊急事態の解決)
- **会計参照:** 0-6 の `M_A(t)`, `Treasury_In/Out`, `Seigniorage_A(t)` に従う。
- **緊急タスク (タイプA):** 1秒を争う事態。「最高値から開始し、時間とともに報酬が減衰する」。誰よりも早く動く者に最大の名誉と報酬を。
- **インフラ・3Kタスク (タイプB):** 継続的な保守。引き受け手がないと報酬が徐々に上昇するが、「実績者への優先通知枠」によりチキンレースを阻止。
- **支払原資 (Soil Treasury):** 公的インフラ報酬は **Seigniorage（明示ルールに基づく新規発行）** と Treasury 収入で賄う。  
  「Burnが収入になる」誤解を避けるため、減価量は支払原資ではなく *指標* として扱う。
- **支払ウォーターフォール:** Treasury 支出が競合する場合の優先順位を明記し、縮退条件をセットで定義する。  
  1) UBI（未受領枠含む）/ Survival Buffer  
  2) 生命・安全に直結する緊急支援  
  3) 既に確定した報酬債務（承認済み掟）  
  4) Targeted Support  
  5) その他の公的支出・実験的補助  
  **縮退条件:** Reserves が閾値を下回った場合、4)以降を自動縮退または停止する。  
  **初期値（暫定）:** Reserves が「直近30日平均支出の2〜4週間分」を下回った場合に縮退。

#### 2. 人間の限界と技術投資プロトコル (お金で解決できない領域)
- **仕様:** 報酬がシステム規定の上限 (キャップ) に達しても誰も引き受けない「高リスク・高負荷」なタスクは、自動的に「人間による実行禁止」フラグを立てる。
- **投資の転換:** その高騰した報酬は、人間への賃金ではなく、「その作業を自動化・ロボット化するための開発費」へと自動的に再配分される。
- **哲学:** 人間の命を切り売りさせない。技術が追いつかない場合は、そのインフラを「閉鎖・放棄」する選択をシステムが促し、人間をシステムへの生贄にさせない。

#### 2-1. 発行上限と人間保護の優先順位（明記）
- **基本原則:** B の発行上限は「安全装置」であり、人間の生存・健康を犠牲にするための予算制約ではない。
- **不足時の扱い:** 予算不足で人が集まらない場合は「停止/保留/自動化への転換」を優先し、ブラック化を容認しない。
- **挑戦への保障:** 困難な挑戦は「誠実なプロセス」自体にも報酬根拠を与え、失敗を罰しない。

#### 3. 「余白」としての無条件UBI
- **仕様:** UBI_base（無条件配布）は、誠実スコアや労働実績に一切紐付けない。
- **受取モデル:** 日次で発生した分は未受領枠として蓄積でき、日次/週次/月次の任意タイミングで一括請求できる。未受領枠は30日まで減価なし、31日以降は通常の減価率を適用する（**発生日基準で一括減価**）。
- **ガス最適化（実装準拠）:** 長期間未請求の場合は `accrueUBIBatched` / `claimUnclaimedBatched` で **日数を分割処理**できる。
- **UI導線（実装準拠）:** 直近の未受領残高を日別に表示し、`accrueUBI` → `claimUnclaimed` の順で一括請求できる。
- **目的:** ニーチェの超人になれない「疲れた時」でも、最低限の生存を保証する。
- **効果:** 報酬がいくら高くても、嫌な仕事は「嫌だ」と言える自由 (拒否権) の根拠を、金銭的な不安を取り除くことで担保する。

#### 4. 労働時間の搾取と「無茶な押し付け」の可視化
- **仕様:** 仕事の「掟 (契約)」に、あらかじめ想定工数と休憩・休日ルールを明記。想定を大幅に超える稼働や短納期を強いた場合、それが「不備報告 (Issue)」として記録され、依頼者の「エンプロイヤー (雇用主) スコア」が低下。
- **効果:** 依頼者の「過去の振る舞い」が全公開されるため、無茶な押し付けをする人のもとには、誰も (UBIがあるため余裕のあるワーカーは特に) 集まらなくなる。

#### 5. 「成功率・定着率」のオンチェーン統計
- **仕様:** 依頼者ごとに「何人が挑戦して何人が断念したか (離脱率)」や「平均継続期間」を自動集計。
- **効果:** 「やる気があるように見せて資金を集めるが、実際は使い捨てが激しい」プロジェクトがひと目で分かり、ワーカーは事前に防衛できる。

#### 5-1. 多重下請け・中抜き対策（透明化と配分固定）
- **直接支払い:** 報酬は掟コントラクトから「実際の作業者アドレス」へ直接送金し、任意の中抜きを原理的に防ぐ。
- **配分比率の固定:** 掟テンプレ登録時に、作業者/管理者/監査者などの分配比率を確定し、後からの恣意的変更を禁止する。
- **例外変更の条件:** 比率変更は全シェア保持者の署名、またはガバナンス承認＋タイムロックを必須とする。
- **手数料の明示:** マネジメント料・仲介料などは掟に明記され、全参加者が事前に確認可能とする。
- **実質下請けの可視化:** 支払い経路と分配比率を監査ログで公開し、過度な中抜きはスコア低下の対象とする。
 - **スコア連動:** 中抜き率が高い、または分配が偏る依頼者は「エンプロイヤー・スコア」が低下し、通知優先度や募集の可視性が下がる。

#### 6. 「完成責任」から「誠実なプロセス (時間)」へのシフト
- **仕様:** 仕事の成果だけでなく、「合意された手順を正しく実行した時間」に対して報酬を支払う (時間での区切り)。不可能の価値として「手順を踏んだが、構造上実現不可能である」と早期に発見・報告した者には、完成報酬に近い額を支払う。
- **効果:** 「知らなかった」「隠して最後までやらせる」といった不誠実なマネジメントを撲滅し、早期の構造見直し (デバッグ) を促す。
- **補足:** 高難度・不確実案件では「誠実な失敗」も価値として評価し、挑戦コストを社会で分担する。
- **状態遷移（最小）:** `Issue -> {Accept | Dispute} -> Resolve`  
  Dispute は DAO 裁定へ移行し、Resolve は二段階確定（Proposed -> Finalized）と再審請求（1回）を許容する。
- **状態機械（最小）:**  
  `None -> Open -> {Accepted | Disputed}`  
  `Disputed -> Proposed -> Finalized`  
  `Finalized -> ReopenedOnce`（再審は1回のみ、条件付き）  
  **不変条件:** Finalized 後の支払いは原則不可逆。再審は「追加補填/別建て清算」で扱い、巻き戻しは避ける。
- **発火条件（最小）:**  
  - `Open -> Accepted`: 作業完了の提出 + 依頼者承認  
  - `Open -> Disputed`: 期限超過/不備報告/依頼者異議  
  - `Disputed -> Proposed`: 裁定案が提示された時  
  - `Proposed -> Finalized`: 確認トランザクション完了  
  - `Finalized -> ReopenedOnce`: 再審請求が期限内かつ条件を満たす場合のみ
- **支払方式の標準化:** 掟の支払は、取引リスクに応じて以下の標準方式を選択する。  
  1) **即時払い:** 低額・低リスクの案件  
  2) **エスクロー:** 検収/段階払いに連動  
  3) **遅延確定:** 争点が多い案件は Resolve 確定後に支払い  
  **チャージバック:** 不備・詐欺が確定した場合はエスクローから返金し、過誤払いは回収ルールに従う。
- **不当Dispute対策（実装方針）:** 受領後の難癖による返金悪用を抑止する。  
  - **適用範囲:** 即時払い/エスクローに適用。遅延確定は支払前のため「保留」ではなく承認ゲートで制御する。  
  - **自動保留（即返金なし）:** Dispute 発生時点で支払額の **20%〜40%** を自動保留し、即時返金は行わない。  
  - **短い確認ウィンドウ:** **48時間**以内に簡易説明（1行）か追加情報がない場合は自動却下。  
  - **自動解放:** 売り手が **48時間** 無反応の場合は保留分の一部（例: **20%**）を自動返金し、残りは通常解放。  
  - **証拠は任意だが有利:** 証拠がある場合は返金率上限を引き上げ、ない場合は下げる（泣き寝入り回避と乱用抑止の両立）。  
  - **少額取引の返金上限:** 低額案件は Dispute でも**最大返金率を30%程度**に抑え、全面返金狙いを防ぐ。  
  - **履歴連動:** Dispute 乱発者は保留比率が上がる／返金上限が下がる。売り手側の不備率が高い場合は逆方向に補正する。
  - **運用手順（最小）:**  
    1) Dispute 発生時に自動保留を適用し、理由/証拠の提出有無を記録  
    2) **48時間**以内に追加情報がない場合は自動却下  
    3) 売り手が **48時間** 無反応の場合は保留分の一部を自動返金  
    4) 高額案件のみ外部裁定へ移行（低額は内部フローで完結）  
    5) 乱用傾向が見えた場合は保留率/返金上限を自動的に厳格化
- **不備報告の設計:** 作業者は救済率を 0〜100% の範囲で請求できる（作業未着手の 0%、依頼側の重大瑕疵による 100% まで許容し、状況の幅を担保する）。依頼者は Accept / Dispute を選び、Dispute は将来の DAO 裁定へ移行する前提で状態遷移のみ行う。悪質性が認定された場合のスラッシングは、知識の非対称性による欺瞞を抑止するための枠として用意し、後続フェーズで裁定ロジックを接続する。
- **外部裁定のプラグイン方針:** `disputeResolver` は外部裁定（Kleros 等）へ接続可能なソケットとして残し、初期は最小運用（単一アドレス/限定ロール）で裁定する。高額案件のみ外部裁定へ切り替える前提。
- **外部裁定の運用ルール（最小）:** 高額案件は `disputeResolver` を外部裁定コントラクトへ切替し、結果のみ取り込む。低額案件は内部解決（自動保留・返金上限）で完結させ、外部手数料を抑える。
- **UXメモ（実装準拠）:** 争議は「Reported → Disputed → Proposed → Resolved」の順で二段階確定。  
  UI は各ステップを明示し、証拠URLはリンク表示で確認できること。

括弧内は正式用語（監査・仕様上の正式名称）を併記する。

#### 7. 「良いものを長く作る」ことへのインセンティブ
- **仕様:** 作ったものが「買い替え」ではなく「長く使われ続けている」間、その作成者に Soil Treasury (公的プール) から「維持報酬」が少しずつ支払われ続ける。
- **減価償却の可視化:** 製品・コードがどれくらい持つのか (耐久年数) を予測値として記録し、実際の寿命がそれを超えたら「誠実スコア」にボーナス。
- **効果:** 作成者は「一度売って終わり」ではなく「長く愛用されるほど得をする」ようになり、質の高いモノづくりが経済的に正当化される。

#### 8. 情報の非対称性の解消 (売る側が隠したい情報の開示)
- **仕様:** 製品やサービスに、耐久年数、過去の修理歴、減価償却の推移をNFT (またはメタデータ) として紐付ける。
- **効果:** 「実はすぐ壊れる」「メンテナンス性が最悪」といった、売る側が知られたくない情報が買い手に筒抜けになる。

#### 9. 資源の流動性と独占防止 (Resource Liquidity)
- **仕様:** 土地や重要インフラなどの物理的資源は、所有者が価値を自己評価して公開し、その評価額に応じた「土壌還流税 (Token B)」を Soil Treasury へ継続的に納める。
- **Harberger Tax:** 高く評価すれば税負担が増え、低く評価すればその価格で他者に買い取られる。資源の価値付けを常に市場と整合させる仕組み。
- **供託金モデル:** 資源の保持時に「立ち退き費用・保証金」を Token B でロック。引き渡し拒否や裁定違反があれば没収し、誠実スコアに強いペナルティを与える。
- **二段階執行:** オンチェーンでは利用権や収益受取権を即時移転し、オフチェーンでは法オラクルを通じて登記や物理的占有の執行を要請する。
- **効果:** 富の集中を物理的にコスト化し、資源が常に社会の最適な場所へ供給されるよう促す。

#### 10. 生産性に基づく通貨価値 (Productivity-backed Value)
- **会計参照:** 0-6 の `M_B(t)` に従う。
- **仕様:** Token B の発行は、検証済みの「実社会への付加価値 (インフラ復旧、良質な製品の提供)」に連動する。
- **検証モデル:** 「Proof of Process (手順の証明)」を基本とし、署名付きログやセンサーデータを根拠に自動判定する。異議申し立てがあれば分散型の事後監査を行い、不正が認められた場合は預託金を没収 (スラッシュ)。
- **効果:** 通貨発行が社会の生産性と同期し、無秩序なインフレを防ぎつつ、外部通貨 (円/ドル) に対する信頼性を担保する。

#### 11. 累進的減価と生存バッファ (Progressive Decay & Survival Buffer)
- **会計参照:** 0-6 の `d_t`, `SB_t` に従う。
- **仕様:** 最小生活費に相当する基本残高 (Survival Buffer) には減価を適用せず (0%)、それを超える余剰資産に対してのみ動的な減価率を適用する。Survival Buffer は固定値ではなく、仕様12/12-1の指数が算出した「最低生活コスト」に連動する動的値とする。
- **一意性:** Survival Buffer の適用は World ID で認証された「プライマリアドレス」に限定し、未認証アドレスは残高に関係なく減価対象とする。
- **初期パラメータ例:** Survival Buffer は「APPIの30日移動平均」で算出し、更新頻度は週次、更新幅の上限は±3%/週とする。
- **効果:** マクロ経済の安定化 (インフレ抑制) のコストを生活者が負うことを防ぎ、富の再分配と通貨循環を両立させる。

#### 12. 購買力保護オラクル (Purchasing Power Oracle)
- **会計参照:** 0-6 の `APPI_t`, `UBI_base(t)`, `UBI_total(t)`, `d_t` に従う。
- **仕様:** APPI（内部CPI）を主指標とし、外部の物価指数 (CPI 等) は補助指標として監視する。Token A の配布額と Survival Buffer の基準値を自動調整する。Token B の発行ロジックとは分離する。
- **供給制御:** 配布増額と同時に減価率も連動して引き上げ、供給と回収を同期させる。短期的な暴走を防ぐため、変化率の上限・下限とスムージング (移動平均) を設定する。
- **初期パラメータ例:** APPI入力は30日移動平均、Token A配布量の調整幅は±5%/週、減価率の調整幅は±2%/週、異常値は3σで除外する。
- **効果:** インフレ下でも、UBI が保証する最低限の生活水準が実質的に目減りしないようにする。

#### 12-1. 自律型購買力インデックス（APPI）
- **目的:** 外部URL依存を避け、システム内データで購買力を算出する。
- **算出方法:** 生存枠バスケット（食料・水・住居・エネルギー等）の掟の成約価格中央値を集計し、内部CPIとして用いる。
- **最低限カテゴリ:** 食料（主食/タンパク源）、水、住居（最低限の居住コスト）、エネルギー（調理/空調/照明）、通信（掟履行に必要な帯域）。
- **検証:** Proof of Process によって承認された取引のみを母集団とし、異常値は自動除外する。
- **外部参照の位置づけ:** 外部CPIは補助指標として用い、内部指数と乖離が大きい場合は監査対象とする。

#### 13. 緊急時前借りプロトコル (Emergency Advance Protocol)
- **会計参照:** 0-6 の `EA_t` と `AdvanceCap_B(t)` に従う。
- **仕様:** 災害などの緊急時に、Token B の将来報酬を前借りする形で一時的に発行できる。発行と同時に、将来の自動回収スケジュールがスマートコントラクトによって確定される。
- **回収制御:** 回収は固定期限ではなく、指標の安定域復帰に応じて動的に実行する。過熱局面では回収速度を上げ、逆風時は緩める。Token A の配布量は仕様12/12-1の制御に一任する。
- **上限設定:** 前借りできる発行量には上限または比率制限を設け、乱用を防ぐ。
- **非常時優先:** 生命・安全に直結する緊急時は「人間保護」を最優先とし、上限の一時緩和はガバナンスの非常時モードで可能とする。
- **効果:** 短期的な危機を救いつつ、中長期的な通貨価値の安定 (Mean Reversion) を担保する。

#### 13-1. 保険プールと再保険（Risk Pools）
- **一次保険:** 小〜中規模の事故・損害をコミュニティ内プールで吸収する（拠出は Token A を基本とし、給付は A またはサービスで支払う）。
- **再保険:** 大規模災害は別プール/外部助成（External_In）と連携し、破綻リスクを分散する。
- **不正対策:** Proof of Process と監査ログにより、保険金詐欺や逆インセンティブを抑止する。

#### 14. 誠実な困窮者への動的支援 (Targeted Support for Integrity)
- **会計参照:** 0-6 の `TS_t` と `UBI_base(t)` に従う。
- **仕様:** 「30日間の平均資産残高が閾値以下」かつ「誠実スコアが一定以上」のユーザーに対し、翌日の配布量を加算する。平均資産残高の計算には、Token A と Token B の合算値 (Token B は時価換算) を用いる。Token B の評価額は、分散型オラクル (Chainlink 等) による「30 日間の時間加重平均価格 (TWAP)」で算出する。
- **判定の平準化:** 瞬間的な残高ではなく移動平均を用いることで、意図的な資産移転による受給 (フラッシュ・ポバティ攻撃) を防止する。
- **不透明な消失の定義:** 掟 (契約) や認証済み決済を通さない、オンチェーン上での「使途不明な大量送金・バーン」が短期間に集中した場合は対象から除外する。
- **公共性の判定:** 支援対象の「正当支出」は、掟テンプレートの公共性評価に統合し、ガバナンス投票だけでなくオンチェーン指標 (利用実績、反証率、分散度) を組み合わせた自動判定にする。指標は最小セットから開始し、監査ログで透明化する。
- **指標の最小セット例:** 30日間のユニーク利用者数、完了率、異議申立率、却下率、上位5主体の利用集中度 (HHI)。
- **異議申し立て:** 判定に対する異議は、ZKP (お天道様システム) による証跡提示でプライバシーを守りつつ DAO で行えるようにする。
- **効果:** 自堕落な浪費を助長せず、真に「まっとうに生き、かつ苦境にある人」をピンポイントで救済する。現実世界の「マイナスの所得税」や「還付金」を、事務コストゼロのリアルタイム処理として代替する。

#### 15. 不労利息の排除と貢献型投資 (Investment over Interest)
- **仕様:** 資産を預けるだけで増える預金金利は存在しない。代わりに、Token B を特定のタスクや技術開発 (ファーストペンギン) にステーキングし、成果報酬や耐久性ロイヤリティの分配を受ける。
- **評価条件:** 投資対象は、掟テンプレートの評価ハッシュが最低閾値以上であるなど、誠実性の履歴が確認できるものに限定する。
- **リスク:** 掟の違反や不正が認定された場合、ステーキング資産はスラッシュされる。
- **効果:** 資産を停滞させず、社会的価値創造への循環を促す。

#### 16. 誠実性に基づく信用供与 (Integrity-based Advance)
- **会計参照:** 0-6 の `DeficitCap_A(t)` と `UBI_base(t)` に従う。
- **仕様:** 認証済みの参加者は、契約した掟の報酬を上限として、Soil Treasury から利息なしで前払いを受けられる。前払い枠は、誠実スコアに応じて段階的に拡張する。
- **段階例:** 初期 10% → 中位 30% → 高位 50% → 最高位 80% などの段階的な上限を設定する。
- **上限設計:** スコアに対して逓減する上限関数を用い、信用膨張の暴走を防ぐ。
- **持ち逃げ対策:** 掟の不履行が確定した場合、未返済分は将来の UBI (Token A) から自動天引きされる。Survival Buffer は天引きの対象外とし、天引き比率は未収債残高・誠実スコア・景気指標に応じて動的に調整する。上限・下限とスムージングを設定し、債務滞留を防ぐ。
- **初期パラメータ例:** 天引き比率の下限5%/上限30% (日次UBI比)、調整は週次、変化幅は±5%/週。未収債の上限は「契約報酬の120%」を初期値とする。
- **再起の制限:** 返済完了まで新規の前払い機能はロックされ、World ID により回避を防ぐ。
- **効果:** 新規参入者にも最低限の信用枠を提供しつつ、実績に応じた挑戦の拡張を可能にする。

#### 16-1. 清算/再建プロトコル（誠実な失敗の復帰）
- **目的:** 誠実に失敗した参加者が再挑戦できる復帰ルートを提供する。
- **清算:** 債務の上限・ヘアカット・免責条件を定義し、破綻の連鎖を防ぐ。
- **再建:** 一定期間の制限付き再参加（前払い枠・高リスク権限の段階復帰）を許容する。
- **接続:** 入口タスク（Public Task Entry）や監査済みの改善行動により、回復を加速できる。

#### 17. 動的資源割当と生存枠 (Resource Quota & Dynamic Pricing)
- **混雑価格:** 土地・水・エネルギーなどの物理資源は、需要集中に応じて還流税率を自動的に引き上げる。
- **生存枠の保護:** 生活に必要な最低限の利用量は「生存枠」として低コストで保証し、超過分には累進的な Token B コストを課す。
- **対象例:** 土地 (最低限の居住・耕作スペース)、水 (生活に不可欠な量)、エネルギー (最低限の照明・調理・空調)、通信 (掟の履行に必要なデータ量)。
- **採取枠の掟:** 漁業・採掘などの採取系資源は、資源量と再生速度に基づいた割当枠を設定し、枠超過はスラッシュ対象とする。
- **算定周期:** 資源の性質に応じて算定周期を変える。水や電力は日次/週次、漁業や林業は繁殖期・成長サイクルに合わせて四半期/年次とする。
- **再生インセンティブ:** 資源の維持・改善が証明された場合、還流税の減免や Token B 報酬を付与する。
- **物理ボトルネック対策:** 重要設備や部品の寿命・故障率・供給途絶を監視し、供給リスクが高い領域には修理・代替・再資源化の報酬を優先配分する。
- **早期警告の評価:** 資源枯渇や供給途絶の予測を提出し、根拠と妥当性が検証された場合は誠実スコア/Token B を付与する。
- **検証プロセス:** 予測の根拠データ（センサー/統計/監査ログ）を提示し、第三者レビューとDAO監査で妥当性を判定する。
- **更新・代替への優先配分:** 修理が非効率な場合は、撤去・更新・新技術導入への転換を支援し、移行コストを公的に補助する。
- **補助条件:** 代替による資源消費削減や安全性向上が示せる場合に限定し、補助は期間上限（例: 6〜12ヶ月）と総量上限（例: 対象資産価値の5〜15%）を設ける。
- **給付型の移行支援:** 「借金」ではなく、将来の損失回避を買い取る形で返済不要のグラントを支給する。
- **撤退・廃棄の完遂ボーナス:** 修理不能な設備を安全に撤去・再資源化した場合、誠実スコアまたは Token B の一時ボーナスを付与する。
- **移行期間のUBI上乗せ:** 大規模な更新/代替プロジェクトを掟として登録した場合、その期間中の Token A 配布量を一時的に増額する。
- **共同負担モデル:** 利用者/受益者が Token B をステーキングして移行費用を分担し、成功時は優先利用権や誠実スコアの加点を得られる。
- **効果:** 共有地の悲劇と事業者の「獲り急ぎ」を同時に抑制し、資源の持続可能性を確保する。

#### 18. 還流の相互共済化 (Reciprocal Reinvestment Buffer)
- **会計参照:** 0-6 の `B_B(t)` と Treasury 収支に従う。
- **仕様:** 停滞資産の還流 (Social Reinvestment) 実績を「救済クレジット」として永続保持する。還流率は一律免除ではなく、稼働中・掟でステーキング中・長期放置などの状態に応じたグラデーション (割引率) を適用する。
- **救済の条件:** 医療・被災・介護・再挑戦などの公的基準に該当した場合、救済クレジットに応じた UBI 加算や緊急支援枠を優先適用する。
- **初期パラメータ例:** 稼働中70%割引、掟ロック中50%割引、長期放置0%割引。割引率の更新は月次、変化幅は±10%/月。
- **効果:** 還流を「没収」ではなく「将来のバックアップ」として再定義し、困窮時の回復力を高める。流動性枯渇を防ぐため、割引率には上限・下限とスムージングを設定する。

#### 19. 流通量ベースの価値安定化 (Circulation-based Stability)
- **会計参照:** 0-6 の `Circulating_B(t)` に従う。
- **仕様:** 価値安定の主要指標を B の総発行量ではなく、ロックされていない流通量 (流通速度/Velocity) に置く。
- **流通からの分離:** 掟へのロック、長期ステーキング、耐久性ロイヤリティなどで固定された B は流通量から除外する。
- **効果:** 社会のインフラや知恵が積み上がる限り総量の増加を許容しつつ、通貨価値の安定を維持する。

#### 20. 参考価格帯の提示 (Guided Price Corridor)
- **仕様:** 価格は当事者が自由に決められるが、過去の類似掟の履歴や現在の通貨状況から「参考価格帯」を提示する。
- **非強制:** 参考価格帯は強制せず、逸脱自体を罰しない。透明な情報提供に徹する。
- **効果:** 価格統制を避けつつ、極端な高騰・不当な低価格のリスクを緩和する。

#### 21. 休息と可処分余裕の制度化 (Rest & Disposable Buffer)
- **仕様:** 掟の報酬設計に「休息・有休・回復」のコストを組み込み、可処分所得の余裕を制度として前提化する。
- **目的:** 休むことを怠惰と見なさず、生活の再生産とメンタル安定に必要な正当コストとして扱う。
- **効果:** 支出を「変換」せずとも、回復のための余裕が構造的に確保される。

#### 22. 資産の社会的還帰と継承 (Nutrient Return & Inheritance)
- **仕様:** 長期間の不活動により休眠と判定された資産は、段階的に Soil Treasury へ還帰する。稼働中・掟でロック中の資産は還帰率を割引し、放置資産には標準率を適用する。
- **初期パラメータ例:** 休眠判定は365日無活動、還帰率は月次で0.5%から開始し、最大2.0%/月まで段階化する。
- **継承の掟:** 生前に継承先を指定でき、通知・猶予・復帰条件を明示して「没収感」を抑える。
- **効果:** 死蔵による循環停止を防ぎ、世代間の資源循環を維持する。

#### 23. 分散型市場観測とデータ報酬 (Distributed Market Witness)
- **仕様:** 参考価格帯の算出に必要な物価・相場データの報告者に、正確性が検証された場合の報酬を付与する。
- **不正対策:** 複数ソース合意、反証可能性、軽微なペナルティを組み合わせて操作耐性を確保する。
- **効果:** 中央集権に依存せず、最新で信頼できる価格情報を持続的に供給できる。

#### 24. 結晶化（Token A → Token B）の例外化と抑止設計
- **会計参照:** 0-6 の `P_AB(t)` と `fee_AB(t)` に従う。
- **位置づけ:** 結晶化は「掟の成果承認」に限って許可される例外的な変換であり、貯蔵回避の抜け道としては利用できない設計とする。
- **自己取引の抑止:** creator と worker が同一（または実質的に同一とみなされる）場合、結晶化は無効化するか、結晶化率を極端に低くする。
- **循環取引への備え:** 複数アドレスの共謀や循環取引は、オンチェーンだけで完全検知できない前提を明記し、監査ログと取引履歴を第三者（公認会計士など）が検証可能な形で参照できる設計を用意する。
- **公開識別の対象:** 公人・公共案件に加え、社会的影響が大きい法人・組織は公開識別の対象とし、公式アドレスや登記情報に紐づく識別子を使用する。
- **同一性検出の扱い:** 実質的な同一性の検出は、循環取引の「確定判定」ではなくリスクスコアの一要素として用い、監査対象の優先度付けに活用する。
- **動的手数料:** 結晶化手数料は固定ではなく、Token A の減価率・市場状況・短期変動に応じて動的に上げ下げする。
- **経済的無利得の保証:** 結晶化コスト（手数料＋機会損失）が、Token A の自然減価による損失を下回らないよう設計する。
- **パラメータ管理:** 結晶化率・手数料の上限/下限および変更権限はガバナンスとタイムロックにより管理する。

### 知財・貢献
#### 1. ファーストペンギン・プロトコル (手順の知的財産化)
- **仕様:** 誰かが「新しい手順 (掟のテンプレート)」を公開し、それが「まっとうな仕事」として成功した実績が増えるほど、そのテンプレートの「評価ハッシュ」が向上。
- **ロイヤリティ:** 他者がそのテンプレートを引用して契約を結ぶ際、成約手数料の一部が自動的に「オリジネーター (開発者)」へ還元。
- **効果:** 「より安全で、より効率的な手順」を考案すること自体が永続的な資産 (Token B) を生む仕組み。

#### 2. 再帰的ロイヤリティと系譜追跡 (Lineage & Recursive Royalty)
- **仕様:** すべての掟 (テンプレート) は登録時に類似度チェックを受け、一定以上の一致がある場合はフォークとしてタグ付けされる。
- **フォーク還元:** フォーク認定された掟が使われた場合、収益の一定割合がオリジネーターへ自動還元される。
- **インポート還元:** 他者の掟を部品として組み込んだ場合、利用料が上位層へ自動送金される。
- **精度補強:** 類似度の一致だけでなく、実運用での利用実績や評価ハッシュを加味して派生判定を行う。
- **効果:** 盗用や車輪の再開発を経済的に無意味化し、正当な積み上げを促進する。

#### 3. コントリビューター・シェア (Contributor Shares)
- **仕様:** 複数の作成者が関わる掟は、登録時に報酬の分配比率 (Shares) を確定させる。
- **変更条件:** 比率の変更には既存の全シェア保持者の署名を必須とする。
- **効果:** 後出しの不当要求を防ぎ、貢献度に応じた報酬を保証する。

#### 4. トロール防止と公知化プロセス (Anti-Troll & Public Domain)
- **ロイヤリティ・ステーキング:** 掟をロイヤリティ対象とするには Token B の預託を必須とし、無意味な量産をコスト化する。
- **公知化の申し立て:** 当たり前の手順に対し DAO が公知化を宣言できる。申し立てにはステーキングを要し、悪用を抑止する。
- **ロイヤリティ減衰:** 先駆者報酬は登録からの時間経過で自動的に減衰し、最終的にパブリックドメインへ還流する。減衰速度はガバナンスで調整可能とする。
- **分配キャップ:** 単一タスクのロイヤリティ総額に上限を設け、テンプレート単位の上限も併設することで、瑣末な先行者の過剰な取り分を抑える。
- **効果:** 価値なき主張はコスト化され、真の貢献が保護される。

### 教育・再挑戦
#### 1. 育成・リスキリング支援 (Nurture & Reskilling)
- **再挑戦の対象判定:** 公的支援の優先度を決めるため、自動化フラグや需要変動などの客観指標を用いる (挑戦の自由自体は制限しない)。
- **後払い型育成投資:** 高額教育は Soil Treasury が先行支援し、成功後の報酬から一定割合を還流する (上限・免除条件を設定)。
- **学びの共有報酬:** 公開された手順や教材が実運用で成果を出した場合、作成者に継続的な報酬を分配する。
- **学び直し支援とBの関係:** 学習プロセスの誠実な実施（記録・成果物・試験）にも少額の Token B を付与し、学習期間の不安を緩和する。
- **効果:** 産業転換や高い教育コストによる機会損失を抑え、再挑戦の循環を維持する。

#### 1-1. 移行期の心理コストとソフトランディング
- **問題意識:** 生活水準の固定化（住宅ローン・教育費など）は、UBIだけでは吸収しきれない心理的圧力を生む。
- **目的:** 「最低保証」への転落恐怖を緩和し、学び直しや職種転換の挑戦コストを下げる。
- **設計:** 誠実スコアが一定以上の参加者に対し、一定期間の返済猶予・柔らかい返済枠・学び直し支援を与える。
- **初期値（暫定）:** 猶予期間 90〜180日、支払い上限は日次UBIの10〜30%相当、再評価は月次。
- **根拠:** 生活再設計に必要な最低期間を確保しつつ、長期固定化を避けるため中期レンジと上限を設定。
- **位置づけ:** 逃げ得ではなく「誠実実績へのリターン」として設計し、再挑戦を促す。

#### 2. 誠実実績の入口タスク（Public Task Entry）
- **目的:** 「Bを稼ぐ手段がない人」に、誠実実績の入口を提供する。
- **設計:** 誰でも参加可能で、手順が明確・検証可能なタスクを公的に開放する。
- **運営方式:** Soil Treasury が「雇用主」になるのではなく、コミュニティ評価を通過したタスクのみを自動公開する市場型とする。
- **不正対策:** World ID による本人性、第三者確認、身体性を伴う作業（現地確認など）を組み合わせる。
- **形骸化対策:** 定期評価で「価値が薄いタスク」は停止。B発行は必要性と検証性を満たすものに限定。
- **具体例:** 防災設備の目視点検、地域清掃、見守り・声かけ、公共データの検証、OSSの簡易バグ再現検証。
- **評価指標:** 完了率、第三者確認一致率、再現性、異議申立率、継続的利用実績。
- **報酬上限:** 入口タスク由来の B 報酬は「日次/週次の上限」を設定し、主たる貢献報酬を置き換えない設計とする。
- **頻度制限:** 同一人物の連続受給はクールダウン期間を設け、分散参加を促す。
- **効果:** 誰でも「誠実さ」を積み上げられ、Bが一部の人の独占物にならない。

### ガバナンス・監査
#### 0. 責任の定義（罰ではなく透明性）
- **定義:** 責任とは「失敗の罰」ではなく「異常や不備を共有する義務」として扱う。
- **原則:** 早期報告は小さな報酬、隠蔽は最大ペナルティにすることで、隠すより出す方が合理的になる設計とする。
- **背景:** 失敗が破滅と結びつくほど隠蔽が増えるため、インセンティブ構造を逆転させる。
- **早期報告の報酬:** 不備報告 (Issue) を適時に出した作業者/管理者に、少額の Token B または誠実スコアの加点を付与する。
- **隠蔽判定の基準:** 事後の監査で「報告すべき異常が存在した証跡（ログ・証拠URI）が確認されたのに、期限内の報告がなかった」場合は隠蔽とみなす。

#### 1. 掟のデバッグ権と責任の所在 (誠実さの再定義)
- **仕様:** 経営者の提示した手順 (掟) に対し、作業者は「不備報告 (Issue)」を出す権利を持つ。
- **強行ペナルティ:** 報告を無視して経営者が「強行 (Force Execute)」した場合、その後の事故や不備の責任 (預託金没収) を経営者が100%負うフラグがオンになる。
- **責任の固定:** Force Execute が発動した時点で、以降の損失・スラッシュは経営者側に固定され、作業者側の預託金は保護される。
- **哲学:** 誠実さとは「言われた通り動く」ことではなく、「不備を隠さず、透明性を保つ」こと。不当な命令に従わなかった記録も「誠実な行動」として評価される。

#### 2. トレードオフ付きガバナンス (Trade-off Governance)
- **仕様:** 最低生活費の定義や配布額の変更を提案する際、システムは自動的に「必要な財源 (減価率の向上)」または「許容されるインフレ率 (為替影響)」のシミュレーション結果を提示する。可決にはコスト負担の合意もセットで必要とする。
- **効果:** 財政的な裏付けのないポピュリズム的決定を構造的に排除する。

#### 3. 監査・検証の継続性 (Continuous Auditability)
- **仕様:** 主要コントラクトの状態とルール適用結果を、誰でも検証可能な監査ログとして公開する。
- **対象例:** Soil Treasury の流入/流出、Token A 配布量、減価率の変更履歴、前払い残高の総量を定期的に記録する。
- **自動検査:** 重要指標の逸脱や不整合を検出した場合、自動的にアラートを発報する。逸脱の目安は直近30日平均からの±20%超など、ガバナンスで調整可能とする。
- **効果:** 「仕様と実装のズレ」を早期に発見し、運用リスクを抑える。

#### 4. 多層的アップグレード・アーキテクチャ (Multi-layer Upgradability)
- **コア・プロキシ:** Soil Treasury や配布ロジックなどの基幹部分はプロキシパターンを採用し、データを保持したままロジック差し替えを可能にする。
- **方式選定:** UUPS を採用し、実行コストの低減とアップグレード規約の柔軟性を確保する。
- **モジュール分離:** 掟の検証や資源価格決定などの周辺機能は独立モジュール化し、機能ごとの改善を容易にする。
- **タイムロック:** アップグレード実行には 30 日の待機期間を必須とし、内容確認と退出の猶予を確保する。
- **権限設計:** アップグレードの実行権限は、誠実スコアが一定閾値以上の参加者から無作為抽出された署名者によるマルチシグ、または DAO 投票に限定する。
- **署名閾値:** 重要操作は選出された 9 名の署名者のうち 5 名以上の合意 (5 of 9) を必須とする。
- **選出方式:** 署名者はランダム選出とし、閾値や抽選ロジックはガバナンスで調整可能とする。
- **透明性:** 署名者の選出プロセスと署名結果は監査ログに記録し、全ユーザーが監視可能とする。
- **設計意図:** 上位者の固定化や少数派の排除を避け、合理性だけに依存しない参加の余地 (近代的自我の限界) を確保するため、閾値 + 無作為抽出と委託の撤回性を採用する。
- **買収・脅迫耐性の配慮:** 完全防止ではなく、買収・脅迫の成立コストを高める設計を重視する。高リスク決議では、匿名化や追加審査などの防衛策を段階的に適用する。
- **委託の安全性:** 委託はいつでも解除・変更可能とし、長期拘束にならないよう設計する。運用負荷が大きい仕組みは検証フェーズを経て段階導入する。
- **署名閾値の可変性:** 重要度に応じて閾値・母集団を調整可能とする。通常変更は 2/3 前後 (例: 7 of 11 または 14 of 21)、重大変更は 2/3〜3/4 と長めのタイムロック、緊急停止は 5 of 9 を想定する。
- **不参加の扱い:** 期限内に反応がない場合は自動棄権とし、補欠署名者で補う運用を想定する。
- **棄権・委任:** 委任は期限付き・いつでも撤回可能とし、棄権は正当な選択として扱う。
- **参加頻度の階層化:** 重要決議は広く、日常的変更は代表 + 監査ログで実施する。
- **再審議条件:** 新情報や重大な不利益が判明した場合に限り、再審議のトリガーを定義する。
- **価値最大化への偏りの抑制:** トークンの資産性や配当最適化よりも、生活安全・誠実性・手続きの正当性を優先することを明記する。
- **参加の健全性:** 委任の集中や参加率の低下をリスクとして監査し、分散性の維持をガバナンスのKPIに含める。
- **動的調整:** これらの数値は、仕様 2 のガバナンスを通じて調整可能とする。
- **移行基準:** ストレージ衝突や基盤チェーンの消失など、プロキシ修正が不可能な状況に限りマイグレーションを許可する。
- **移行手段:** マイグレーション時は手順を公開し、タイムロックと退出権 (仕様4) によりユーザーの選択を保障する。
- **効果:** 修正可能性と透明性を両立させ、長期運用に耐える構成を維持する。

#### 5. 報いの権利化 (Merit as Rights, not Balances)
- **仕様:** Token B の残高が減っても、過去の誠実な貢献に基づく「権利・特典 (優先権、割引、提示順、レビュー優先などのソフト特典)」は保持される。
- **権利の上限:** 特典は生活や参加を妨げない軽度の優先に限定し、意思決定の独占や排他権には接続しない。
- **集中の抑制:** 単一個人/組織への特典集中を監査し、上限や期限、逓減ルールで偏在を防ぐ。
- **効果:** 報いが「残高の大小」ではなく「履歴と信頼」に保存され、使うほど損だと感じる心理を抑える。

#### 6. 権限の鮮度重視 (Recency-weighted Authority)
- **仕様:** 投票権・裁定権などの意思決定権限は「直近の活動」を重く評価し、永久スコアは名誉・履歴として軽く反映する。
- **活動減衰:** 一定期間の不活動は権限を自動的に逓減させ、再参加によって回復できる。
- **権限の上限:** 単一主体の過大な集中を避けるため、影響力の上限と冷却期間を設定する。
- **効果:** 権限の固定化を防ぎ、「いまの状態で判断できる人」に決定権を配分する。

#### 6-1. 信用スコアの安全設計（不可侵領域・上限・回復）
- **不可侵領域:** UBI、Survival Buffer、異議申立て、再審、退出権はスコアで制限しない。
- **影響上限:** 提示順・優先枠・前払い上限などの影響は逓減関数で頭打ちにする。
- **用途別スコア:** 安全遵守/支払い誠実/公共貢献などに分離し、単一スコアで全領域を支配しない。
- **回復ルート:** 時間減衰と入口タスク（Public Task Entry）により、低スコアから再起可能とする。
- **誤判定の保護:** 証拠の確度が低い場合はスコアを暫定（保留）扱いとし、強い制裁に使えないようにする。
- **許可リスト方式:** スコアを使ってよい用途（例: 掟の検索順位、通知優先、前払い枠、監査優先度）を列挙し、それ以外の接続はガバナンス承認＋タイムロック必須とする。
- **許可リスト初期値（例）:**  
  - 掟の検索順位: 上位表示は最大でも上位10%まで  
  - 通知優先: 1日あたり最大3件まで  
  - 前払い枠: 絶対上限は契約報酬の50%  
  - 監査優先度: 監査対象の優先順位のみで、強制停止には使わない
- **確度（confidence）:** スコア値と証拠確度を分離し、確度が低い場合は資金リスクのある機能に使用できないよう制限する。
- **確度の判定基準:** 署名付きログ、第三者確認一致率、反証率、経時安定性（30日移動平均）などを統合して算出する。

#### 7. 代理設定のガードレール (Delegated Claim Guardrails)
- **仕様:** Token A の受取プロファイルの代理設定は、本人の権利保護を最優先に設計する。
- **有効期限:** 代理権限は上限期間 (例: 90日) を設け、自動失効とする。
- **撤回権:** 本人はいつでも即時に解除・変更でき、拒否権は常に本人が保持する。
- **監査性:** 代理による設定変更は監査ログに記録し、第三者検証可能とする。
- **権限範囲:** 代理権限は「受取頻度/上限の設定」に限定し、送金先の変更や資産移動は不可とする。
- **同意再確認:** 代理の付与・更新には本人の明示同意と再確認を必須とする。

### 運用・安全
#### 0. 想定攻撃と防御（最小）
- **攻撃モデル:** 循環取引/共謀、虚偽証拠、オラクル操作、買収・脅迫、委任悪用、フラッシュ貧困、Sybil攻撃。
- **防御:** 上限設定、監査ログ、証拠確度（confidence）、タイムロック、ZKP、再審、サーキットブレーカー、入口タスクによる回復ルート。
- **対応表（簡易）:**  
  - 循環取引/共謀 → 結晶化上限・監査優先・同一性リスクスコア  
  - 虚偽証拠 → ZKP・第三者確認・confidence  
  - オラクル操作 → フェイルセーフ・外部補助・サーキットブレーカー  
  - 買収/脅迫 → タイムロック・再審・多層署名  
  - 委任悪用 → 期限付き委任・即時撤回  
  - フラッシュ貧困 → 移動平均・不透明消失除外  
  - Sybil攻撃 → World ID・代替IDモジュール

#### 0-1. 未解決課題と緩和策（明文化）
以下は**完全解決が困難**な領域であり、「ゼロにする」ではなく「悪化を抑える」方針で設計する。
- **オラクル問題（現実の真偽）:** 完全検知は不可能。  
  **緩和:** 証拠確度（confidence）と第三者確認の統合、監査ログの公開、リスクスコアによる優先監査、重大案件は追加証拠やオフチェーン再確認を要求。**正直者が必ず勝つ保証ではなく、偽装コストを引き上げる**方向に寄せる。
- **加盟店/売り手の換金圧:** トークンを即時換金したくなる動機は残る。  
  **緩和:** 小さな勝利領域からの段階導入、結晶化の上限/手数料/動的調整、B報酬や長期ロイヤリティ等の**保持インセンティブ**で売り圧を分散させる（運用・制度設計で調整）。
- **プライバシー/監視感:** 透明性が高いほど監視感は増す。  
  **緩和:** ZKP・データ最小化・堆肥化、用途別 nullifier 分離、evidenceHash のみオンチェーン化、オフチェーン証拠の参照分離。**長期の追跡リスクを下げる設計**を優先する。
- **裁定コスト/遅延:** すべての揉め事を人力裁定で扱うのは非現実。  
  **緩和:** 低額・低リスクは即時/自動、争点の大きい案件のみ裁定へ段階移行。二段階Resolve・再審1回・テンプレ化された受入基準で**裁定の総量と遅延を抑制**する。

#### 0-2. 換金圧の緩和（具体策メモ）
加盟店・売り手が「受け取った直後に換金したくなる」前提を置いたうえで、**即売りの単一行動を優位にしない**設計を用意する。
- **初期実装スコープ:** Phase1 は原則「方針/運用メモ」に留め、オンチェーンの必須ルールは結晶化の上限/手数料/動的調整のみ。その他は運用実験として段階導入する。
- **用途限定の即時価値:** 取引手数料の割引、掲載/露出の優先、一定の信用枠（前払い枠）など、**保有・利用するほど得**になる非金銭的メリットを付与する。
- **B報酬/ロイヤリティの遅延付与:** すぐ換金すると失う長期メリットを明示し、**保持期間に比例する報酬**（維持報酬・耐久性ロイヤリティ）を設計する。
- **結晶化のコスト設計:** A→B結晶化は「自然減価より得にならない」範囲の手数料・上限・動的調整を維持し、**短期裁定の魅力を下げる**。
- **売り圧の集中回避:** ガバナンス/運用で、特定期間に換金が集中しないよう**緩やかな上限・クールダウン**を採用する（市場の急変時は例外モード）。
- **小さな勝利からの定着:** 法定通貨決済が必須な領域を避け、**当面はコミュニティ内で完結する支払・評価用途**に寄せて定着させる。

#### 1. 「お天道様 (ZKP)」によるプライバシーと抑止力
- **技術的アプローチ:** ゼロ知識証明 (ZKP) を使用。「具体的な作業の詳細は秘匿したまま、合意された手順A, B, Cを正しく踏んだこと」という数学的な証拠だけをチェーンに記録。
- **心理的効果:** 「常に監視されている窮屈さ」ではなく、「いざ疑われた時に、自分はまっとうであったと証明し、自分を守ってくれる盾 (ログ)」を各自が保持している状態。

#### 2. セーフモードと段階的停止 (Circuit Breaker)
- **仕様:** オラクル異常や重大バグが検出された場合、配布・ミント・回収の一部を段階的に停止するサーキットブレーカーを用意する。
- **発動条件:** 監査ログの逸脱アラート、オラクル値の急変、重大な脆弱性報告が確認された場合に自動発動する。
- **セーフモード時の操作:** 受取・閲覧は許可し、ミントや大口移転などリスクの高い操作のみを段階的に制限する。
- **解除条件:** ガバナンスの承認と監査済み修正が揃った場合にのみ復帰できる。
- **効果:** 異常時の被害拡大を防ぎ、回復性を担保する。
- **危機時の縮退/退出:** 外部強制力や地政学的危機が発生した場合、証拠保存・資産保護・退出支援を優先し、機能を最小限に縮退する。
- **段階の最小定義（実装準拠）:**  
  - **Normal:** UBI発生/請求、APPI反映、結晶化、報酬支払など通常運用  
  - **Limited:** UBI関連（claim/accrue/unclaimed）と結晶化を停止、閲覧は継続  
  - **Halted:** UBI関連・結晶化・緊急発行を停止、閲覧のみ継続

#### 3. レッドチームと脆弱性報奨 (Red Team & Bounty)
- **仕様:** 仕様や実装の脆弱性を報告した者に対し、報奨と誠実スコアの加点を与える。
- **制裁:** 悪用が確認された場合はスラッシュと参加制限を適用する。
- **効果:** 攻撃者を「味方」に変え、早期発見の動機づけを行う。

#### 4. 退出・移行の権利 (Exit & Migration)
- **仕様:** 参加者は任意のタイミングで、保有資産の移転や評価停止を申請できる。
- **清算ルール:** 未払いの前払い・ステーキング・ロイヤリティがある場合は清算を優先し、残余資産のみ移行可能とする。
- **効果:** ロックインを避け、透明で健全な参加継続を支える。

#### 5. 将来不安の低減と透明性 (Future-facing Transparency)
- **仕様:** 予測可能な範囲で、還流率・救済枠・想定残高の見通しをダッシュボードで可視化する。
- **非約束:** 予測は拘束力のない見通しとして扱い、変動要因と前提条件を明示する。
- **効果:** 「未来が悪くなる予測」による不安を減らし、行動計画を立てやすくする。

#### 6. データ最小化と長期プライバシー (Data Minimization)
- **仕様:** 永続記録するデータは最小限に限定し、証跡は ZKP で代替する。
- **保持方針:** 個人情報に紐づくデータは保持期間を制限し、削除・失効の手続きを定義する。
- **効果:** 長期的なプライバシー侵害リスクを低減する。

#### 7. コア資金の保護と権限管理 (Treasury Security)
- **仕様:** Soil Treasury の管理権限は、マルチシグとタイムロックを必須とし、単一鍵の操作を禁止する。
- **運用ルール:** 重要な権限変更は監査ログに記録し、ハードウェア鍵や分散保管を前提とする。
- **効果:** 単一障害点や内部不正による資金流出を抑止する。

#### 8. オラクルのフェイルセーフ (Oracle Fail-safe)
- **仕様:** APPI/CPI/TWAP 等のオラクルが停止または異常値を示した場合は、配布・ミントを凍結し、前回値固定または保守値へ自動切替する。
- **復帰条件:** 正常なデータソースが複数一致し、監査ログが安定した場合のみ解除する。
- **効果:** オラクル破綻による供給暴走を防ぐ。

#### 9. データの堆肥化とステート圧縮 (Data Pruning & State Compression)
- **破棄後の検証性:** 生データは一定期間後に破棄するが、ハッシュ鎖と要約された ZKP により手続きの正当性は永続的に検証可能とする。
- **データのライフサイクル:** 生ログは短期保持 (例: 90日) とし、期限後は証明データのみを残す。
- **アーカイブの役割:** 一般ユーザーは軽量化のために破棄する一方、報酬 (Token B) を得るインデクサー・ノード (記録者) が長期アーカイブを保持し、必要時に検索・提供する。
- **効果:** ストレージ肥大化を抑制し、一般デバイスでも数GB以内の運用を可能にする。

#### 10. 非金利型の資金循環モデル (Non-Interest Financial Loop)
- **リスク分担:** プロジェクトへの投資は成功時の成果分配と引き換えに、失敗や不正時の連帯スラッシング (資産没収) を伴う。
- **動機づけの多層化:** 投資家の見返りは金銭的リターンに加え、誠実スコアの向上 (ガバナンス権の拡大) および支援先サービスの優先利用権として提供する。
- **効果:** 不労所得を排除し、審査と伴走を前提とした能動的な投資行動へ誘導する。

#### 10. 重要データの永続化と凍結 (Data Persistence & Freezing)
- **凍結フラグ:** 紛争、重大事件、歴史的価値があると認められたデータは、自動削除 (堆肥化) の対象外とする。
- **保存先:** 凍結された生データは分散型永続ストレージへ移行し、ハッシュ値はオンチェーンに永続記録する。
- **申請権と強制凍結:** ユーザーは自身の証明のためにステーキング付きで保存申請できる。DAO は不当な隠蔽や重大性が疑われる場合に強制凍結できる。
- **乱用抑止:** 凍結には最低預託量と再審査期間を設定し、悪用や過剰保存を抑制する。
- **効果:** 未解決事件や文脈依存のトラブルでも、長期的な証拠能力を確保する。

## 導入戦略 (Adoption Path)
- **段階導入:** いきなり法定通貨の代替を目指さず、地域通貨・ポイント・評価制度として開始する。
- **小さな勝利:** OSS開発、協同組合、災害支援、自治体の試験運用など、透明性と信頼が重要な領域から実績を積む。
- **UX優先:** World ID やガスレス操作を前提に、参加コストを最小化する。
- **IDの代替性:** World ID 依存を単一障害点にしないため、複数IDプロバイダ/ローカルKYC/コミュニティ証明などの差し替え可能なモジュール設計を前提とする。
- **現実連携:** 物理資源や法的執行は法オラクルと連携し、実社会の枠組みと整合させる。
- **既存勢力の転換:** 排除ではなく、貢献型投資やリスク低減のメリットで参加を促す。

# FairSoil ビジョン（詳細）

このドキュメントは README_ja.md から分離した「ビジョン詳細」です。  
実装仕様は `docs/spec_ja.md` を参照してください。

関連:
- English vision: `docs/vision_en.md`
- 英語仕様（要約）: `docs/spec_en.md`

## ビジョン（要約）
- FairSoil は「誠実さが損にならない」経済基盤を目指す。
- UBI によって、望ましくない仕事を断っても生存が脅かされない状態を作る。
- 社会的に必要だが困難/危険なタスクが、公正なインセンティブで報われる土壌を整える。

## 前提と設計姿勢（「仕方ない」と「仕方なくない」の分離）
- **前提:** 資源や良いポストは有限であり、競争自体はゼロにはならない。
- **設計姿勢:** 「性格の悪い人が必ず勝つ」という状態は**ルール設計で変えられる**。  
  外部化・情報非対称・生存の恐怖によって不正が得になる構造を、可視化・コスト化・UBIによる拒否権で反転させる。

## システム vs 法（境界線の引き方）
- **役割分担:** FairSoil は「正直者が損をしない」「無茶な搾取を記録する」「UBI を配る」というインセンティブの提供に徹する。暴力などの物理的犯罪や土地登記の強制力は現実の法・組織が担う。
- **オラクルとしての法:** 裁判所などの判断結果を外部データとして取り込み、スコアや評価に反映するハイブリッド型で柔軟性を担保する。
  - **取り込み条件:** 最終確定のみ反映し、係争中は保留扱いとする。
  - **検証:** 取り込みフォーマットと署名検証を必須とする。
- **効果:** すべてをプログラムで固定化するディストピアを避けつつ、現実社会との整合性を保つ。

## 適用範囲と限界
- **適用範囲:** FairSoil は法治と合意が機能する範囲で最大効果を発揮する。
- **限界:** 武力・強制が支配する環境では、合意・透明性を前提とする仕組みは本質的に弱い。

## 擬態による実装戦略（厳格なカーネルと親切なシェル）
FairSoil は「社会OS」として、裏側（Smart Contracts）において厳格な経済物理法則を貫く。  
一方で表側（Frontend/UX）では、ポイント、クーポン、サポート窓口など「親しみのある概念」に抽象化する。  
これはユーザーに複雑な理論を強要せず、直感的な操作で「誠実さが報われる」結果だけを受け取れるようにするための設計である。

# FairSoil Phase2+ ノート（日本語）

## Phase2+ 構想（概要）
- APPI の confidence 強化（価格操作耐性）。
- 追加のガバナンス/インセンティブ・モジュール（QF/RPGF、予測など）。
- UX/AI 支援の拡張（オフチェーンのみ）。

## Phase2+ 構想（詳細）
### 0) 外部裁定 / 外部陪審ソケット
- **目的:** 低資産層が dispute で不利になりやすい偏りを弱め、村内の資産格差から裁定を切り離す。
- **背景:** Phase1 の近似シミュレーションでは、deposit 緩和だけでは low-balance 参加者の勝率改善が限定的で、裁定ロジックの独立性の方が効果が大きかった。
- **方針:** `disputeResolver` は将来の外部裁定コントラクトや jury システムへ差し替え可能なソケットとして維持する。
- **裁定原則:** 参照すべきなのは、提出証拠、手順整合性、時系列、過去の悪意履歴であり、当事者の資産量そのものではない。
- **制度設計メモ:** 形式だけ第三者に見えても、当事者兼審判、権限の一極集中、引き延ばしが得になる手続は避ける。詳細は `docs/adjudication_antipatterns_ja.md`。
- **Phase1 の最小仕様:** dispute を重くしすぎず成立させるための最小構成は `docs/phase1_minimal_dispute_spec_ja.md` に分離している。
- **最小インターフェース案:**
  - `requestExternalResolution(covenantId, evidenceRoot, metadataUri)`
  - `finalizeExternalResolution(covenantId, workerPayoutBps, integrityPoints, slashingPenalty, rulingHash)`
- **導入順序:**
  1) Phase1 は単一の `dispute arbiter` が手動運用
  2) Phase2 で高額案件のみ外部裁定へ route
  3) 安定後に jury / elected arbiter / external protocol へ拡張
- **注意:** 低資産側への一律補正をオンチェーン判定に直接ハードコードしない。悪用されるため、補正は外部裁定や手続き設計で吸収する。

### 1) APPI Confidence 強化
- APPI の寄与に confidence を付与（例: 最小報告人数、多様性、鮮度）。
- 相関の高い報告を減衰し、価格操作を抑える。
  - **最小実装:** confidenceBps（0〜100%）と maxReportsPerCategory を追加し、中央値に重み付けを適用。

### 2) 公共財ファンディング・モジュール
- Quadratic Funding（QF）で幅広い参加を促進。
- Retroactive Public Goods Funding（RPGF）で成果確認後に報酬。

### 3) 予測と早期警告
- サプライチェーン等のリスクに対する予測市場/ステーキング。
- 「当てた利益」ではなく、**後に正しいと証明された早期警告**を報酬化。

### 4) 保険・再保険プール
- コミュニティ単位の保険プールで事故リスクを吸収。
- 大災害向けに再保険レイヤーを用意。

### 5) コンプライアンス・モジュール（任意）
- 地域要件に合わせた差し替え可能なアダプタ。
- コミュニティ単位で opt-in（全体強制にはしない）。

### 6) 資源税（Harberger）の最小実装
- **目的:** 資源の独占コスト化を最小構成で実装し、循環を促す。
- **最小仕様:** ResourceRegistry が資源IDを登録し、自己評価額と税率に基づいて Token B を定期支払い（pull型）で Treasury に移転する。
- **買い取り口:** `buyResource` を用意し、評価額以上でオンチェーンの所有権のみ移転（物理執行は後回し）。

### 7) CovenantLibrary（テンプレ共有 + ロイヤリティ）
- **目的:** 良い掟（手順/テンプレ）の共有と再利用を促進し、先駆者へ還流する。
- **最小仕様:** テンプレの登録（metadataUri/royaltyBps）と有効化/無効化、ロイヤリティ計算のみを提供する。
- **注意:** 自動分配は後続フェーズで接続（まずはイベント記録）。
  - **運用メモ:** テンプレ使用は `recordUse` で記録し、後続の分配ロジックに接続する。
  - **接続設計（ロイヤリティ自動分配の最小案）:**
    1) Covenantの決済時（`approve`/`resolveDispute`）に、支払い額と `templateId` を参照可能にする。
    2) `RoyaltyRouter` のような専用コントラクトを後続で追加し、Covenantから `notifyPayout(covenantId, templateId, amountB)` を呼び出す。
    3) Routerは `CovenantLibrary` の `royaltyBps` からロイヤリティ額を計算し、テンプレ作成者へ `TokenB` を送る。
    4) 二重支払い防止のため `royaltyPaid[covenantId]` を保持する。
    5) フォールバック運用として、当面はイベント集計→運用者が `settleRoyalty` を実行する方式でも良い。

### 8) 教育支援テンプレ（Covenant流用）
- **目的:** 後払い型の育成支援を「掟テンプレ」として提供する。
- **最小仕様:** 教育支援用テンプレ（metadataUri）を用意し、Covenant作成時にテンプレIDを指定して `recordUse` を記録する。
- **運用:** 成果判定や返済条件の適用はオフチェーン運用から開始し、後続でオンチェーン化する。
  - **運用フロー（最小）:**
    1) 教育支援テンプレを登録（metadataUriに「対象/期間/達成基準/返済条件」を記載）
    2) 受講者が Covenant を作成しテンプレIDを指定（`recordUse` で記録）
    3) 進捗・成果はオフチェーンで検証（提出物/試験/第三者評価）
    4) 成果達成時に報酬/返済を Covenant の支払として確定
    5) 返済条件は最初は運用で管理し、後続フェーズで自動化

# Phase1 Dispute Fairness（JA）

このメモは、FairSoil Phase1 の dispute fairness 方針を整理したものです。  
特に「低資産層が dispute で不利にならないか」という懸念に対する暫定原則を明文化する。

関連:
- 最小仕様の整理: `docs/phase1_minimal_dispute_spec_ja.md`
- 制度上避けたい失敗: `docs/adjudication_antipatterns_ja.md`

## 背景

Phase1 の近似シミュレーションでは、次が観測された。

- UBI そのものによる即時のインフレ崩壊は強く出ていない
- 一方で low-balance 参加者は dispute で不利になりやすい
- deposit 緩和だけでは偏りは十分に改善しない
- 裁定ロジックの独立性や external arbiter 方向の方が効く

## Phase1 の基本原則

### 1. 裁定は資産量を直接の優劣に使わない

裁定結果は、どちらが多くの Token B を持っているかで決めない。  
資産差は spam 防止や deposit 設計には使っても、勝敗そのものの根拠にはしない。

### 2. まず証拠と時系列を見る

裁定の主軸は次の順で考える。

1. evidence / evidenceUri / evidenceHash
2. 手順整合性
3. 時系列
4. 悪意履歴や cooldown 状況
5. integrity（補助的）

### 3. integrity は補助指標であり、絶対的な正しさではない

integrity は参考にはなるが、共謀や相互評価ポンジを完全には防げない。  
したがって Phase1 の arbiter は、integrity を「証拠より強い根拠」として扱わない。

### 4. defense quota は参入補助であって、勝率補正ではない

defense quota や virtual stake は「低資産層でも dispute を始められる」ための仕組みであり、  
自動的にその側を勝たせるものではない。

## Phase1 の実務ルール

- arbiter は wallet size を判断理由に書かない
- arbiter は evidence と timeline を first-class input として扱う
- 高額案件は可能なら外部裁定へ回す
- UI でも「wallet size ではなく evidence と timeline を見る」と明示する

### Arbiter の最小レビュー記録

Phase1 の arbiter は、proposal 前に少なくとも次の 4 項目を残す。

1. `Claim summary`
   worker が何を求めているか。支払割合や争点を短く要約する。
2. `Requester response`
   requester の反論・反証・手続上の主張を短く要約する。
3. `Missing evidence / gaps`
   まだ不足している証拠、時系列の欠落、矛盾点を書く。
4. `Recommended payout`
   最終的に worker に何 % 支払う提案かを明示する。

これは free text の感想ではなく、後から第三者が「何を見てその proposal に至ったか」を追跡できる最小記録である。

## Phase2 への接続

Phase1 の manual arbiter は暫定運用であり、将来は次へ移行する。

- external arbiter / jury
- elected arbiter
- high-value disputes only routing

## やらないこと

- 低資産側に一律ボーナスをオンチェーンで直接ハードコードする
- 単純に deposit を下げるだけで「公平になった」と見なす

## Phase1 として十分な到達点

- UI 上で裁定原則が明示されている
- arbiter 運用メモが残っている
- シミュレーションで偏りを継続監視している
- external adjudication に差し替える socket を壊していない

# Phase1 Minimal Dispute Spec（JA）

このメモは、FairSoil の Phase1 dispute を**複雑にしすぎず成立させる最小仕様**として整理したものです。  
目的は、完全な裁判制度を再現することではなく、**仕事の踏み倒し・長期引き延ばし・低資産者の泣き寝入り**を減らすための最小防衛線を定義することです。

関連:
- requester 側の最小保護: `docs/phase1_requester_protection_spec_ja.md`
- Scope を削る候補: `docs/phase1_minimal_scope_cutlist_ja.md`
- 公開しやすい dispute record 設計: `docs/dispute_record_publication_ja.md`

## 1. 位置づけ

Phase1 の dispute は、FairSoil 全体の中心ではなく、次のコアを補助するための仕組みです。

- Token A による最低限の生活・交渉拒否権
- Token B による労働報酬の蓄積
- escrow 付きの work agreement

したがって、dispute は「完全な正義の自動化」ではなく、**escrow だけでは吸収できない揉め事を軽く処理するための例外系**として扱う。

## 2. Phase1 で残すもの

Phase1 では、次だけを dispute の必須機能として残す。

1. worker が issue を出せる  
2. requester が accept または challenge を選べる  
3. 第三者が proposal を出せる  
4. 別ロールが finalization できる  
5. evidence URL / hash / short reason を残せる  
6. audit trail に「誰が、どの role で、何をしたか」が残る

これで十分です。  
まず必要なのは、「揉めた時に何も手段がない状態」を避けることであり、法廷級の制度完成ではありません。

補足として、Phase1 の requester challenge は、完全対称の反訴ではなく  
**限定的な misconduct report として再定義する** のがよい。  
詳細は `docs/phase1_requester_protection_spec_ja.md` を参照。

## 3. Phase1 で削る・後ろに送るもの

以下は重要だが、Phase1 の必須条件ではない。

### 3-1. 裁定制度の高度化

- external arbiter / jury の本接続
- elected resolver
- resolver 候補プールの高度選出
- DAO 裁定

### 3-2. dispute UI の過剰な構造化

- 長い arbiter review フォーム
- 細分化された summary / gap / context 項目の必須化
- 小額案件にも重い review checklist を要求すること

### 3-3. 自動化しすぎた公平性補正

- 低資産側への一律有利補正
- 複雑な reputation 数式で勝率を調整する設計
- 「誰が弱者か」をオンチェーンの算術で完全判定する発想

これらは複雑さを増やしやすく、制度への不信や悪用余地を生む。  
Phase1 では、**簡素な手続 + role 分離 + 証拠中心**で十分です。

## 4. Phase1 の最小入力

dispute に必要な入力は、原則として次で足ります。

### worker 側

- claim %（0〜100）
- short reason
- evidence URL または evidence hash

### requester 側

- accept または challenge
- challenge する場合の short reason
- 必要なら evidence URL

### resolver / finalizer 側

- 推奨 payout %
- 最終メモ（短くてよい）

長文の作文や法務的な表現は必須にしない。  
重要なのは、**第三者が後から見て流れを追えるだけの最小情報**です。

## 5. 期限ルール

Phase1 の dispute は、**時間を武器にできないこと**が重要です。  
そのため、最低限次の期限設計を持つべきです。

1. worker issue 期限  
2. requester response 期限  
3. resolver proposal 期限  
4. finalizer confirm 期限

期限切れ時は、何もしないまま無限に止まるのではなく、**default outcome** を用意する。

例:

- requester が期限内に反応しない  
  -> worker claim の範囲で自動採用、または resolver review へ自動遷移
- resolver が長期間未対応  
  -> simple split / refund rule を暫定適用
- finalizer が未対応  
  -> proposal を自動確定、または上位フローへ移送

Phase1 では完全な自動化は不要だが、**放置が最も得になる設計**は避ける。

## 6. 小額案件と高額案件を分ける

Phase1 では、すべての案件を同じ重さで扱わない。

### 小額案件

- 短い理由
- 少ない入力
- 短い期限
- 軽い hold

### 高額案件

- 追加 evidence
- 追加 review
- 将来の external adjudication に接続可能

この分離がないと、低額 dispute でも「勝っても大損」になりやすい。

## 7. “勝っても大損” を防ぐ最低限ルール

Phase1 の dispute は、結論の正しさだけでなく、**争うコストの上限**を意識する。

### 7-1. 入力負担を小さくする

- 長文必須にしない
- URL と短い理由で始められるようにする
- 小額案件では特に軽い入力にとどめる

### 7-2. 再審や蒸し返しを制限する

- 再審回数は 1 回まで
- 同一争点の無限再提出を認めない

### 7-3. challenge に小さなコストを持たせる

- ただし低資産者を締め出さない水準にする
- defense quota や軽減枠は維持する

### 7-4. 不必要な長期凍結を避ける

- escrow は dispute のために全期間無制限で止めない
- hold する範囲と期間を明確にする

## 8. 裁定基準

Phase1 では、次を裁定の主軸とする。

1. 提出証拠  
2. 手順整合性  
3. 時系列  
4. 明らかな悪意履歴  
5. integrity（補助）

`wallet size` は裁定結果の根拠に使わない。  
資産差は spam 対策や担保設計に影響しても、**正しさの根拠そのものにはしない**。

## 9. 第三者性の最小条件

Phase1 の時点で完全な陪審制度は不要です。  
ただし最低限、次は守るべきです。

1. resolver と finalizer を分ける  
2. できるだけ当事者と利害関係を持つ人を外す  
3. role 行使を audit trail に残す  
4. 裁定理由を短くても残す

第三者性は「外部っぽい名前を付けること」ではなく、**当事者・運営・裁定者の癒着を見えやすくし、抑えること**から始める。

## 10. まとめ

Phase1 の最小 dispute は、次の一文で表せます。

> **証拠つきで異議を出せて、相手が応答できて、最後に role 分離された第三者が短い理由付きで終わらせる。**

これ以上の複雑な裁定制度は、Phase1 では必須ではない。  
先に守るべきなのは、次の3点です。

1. 一方的な踏み倒しを防ぐ  
2. 争いを長引かせるだけで得をしにくくする  
3. 正しい側が dispute するだけで潰れないようにする

FairSoil の Phase1 は、この最小防衛線を越えれば十分です。

# Phase1 Requester Protection Spec（JA）

このメモは、FairSoil Phase1 で **worker 側の issue 救済だけが強くなりすぎないための最小仕様** を整理したものです。  
目的は、requester にも最低限の保護を与えつつ、**依頼者の立場を使った圧迫や恣意的な品質攻撃**を制度化しないことです。

## 1. 基本方針

FairSoil の Phase1 では、worker 側の issue は未払い・不当 reject・条件違反に対する救済として重要です。  
ただし、その入口だけが強すぎると、逆に次の偏りが起こり得ます。

- requester が明白な no-show や虚偽提出を受けても、十分な手段を持てない
- 「弱さ」「遅さ」「能力不足」を盾にして、悪意のある不履行が見逃される
- worker 救済の設計が、逆方向の一方的優位になってしまう

そのため、Phase1 では **完全対称の dispute** ではなく、  
`worker issue` に対して、requester 側には **限定的な misconduct report** を持たせるのがよい。

Phase1 の現実的な実装では、これは新しい on-chain state を増やすのではなく、  
**既存の requester challenge UI を misconduct report として再定義する** 形で導入するのが望ましい。

## 2. worker issue と requester report の違い

### worker issue

worker issue は主に次を扱う。

- 未払い
- 不当な reject
- 合意条件違反
- 一方的な条件変更

これは worker を守るための救済手段であり、Phase1 の中心として残す。

### requester misconduct report

requester 側に追加するのは、worker issue と完全対称の「反訴」ではない。  
対象は **能力評価ではなく、手続違反・虚偽・未提出** に限定する。

## 3. Requester misconduct report が扱うもの

Phase1 で扱う対象は、次の4類型に絞る。

### 3-1. No-show

- 合意後、期限までに提出がない
- 連絡や更新もなく、実質的に放棄している

### 3-2. False submission

- 作業未実施なのに、実施済みと偽る
- 明白に虚偽の提出物や説明を出す

### 3-3. Materially incomplete work

- 合意の中心要件を明らかに満たしていない
- 単なる「気に入らない」ではなく、主要部分の未達である

### 3-4. Repeated procedural failure

- 必要な修正応答や説明を繰り返し無視する
- 期限付きの基本手続を守らない

## 4. 扱わないもの

次は Phase1 では扱わない。

- 「なんとなく気に入らない」
- 依頼者の主観的な好み
- worker の性格や能力そのもの
- 遅い、不器用、弱い、といった属性評価
- 最低限の合意は満たしているのに、依頼者の満足だけが足りないケース

FairSoil が裁くのは **人格や能力** ではなく、  
**合意違反と証拠可能な行為** です。

## 5. 入力項目

requester 側の report は、できるだけ軽くする。

### 最小入力

1. `report type`
   - `no-show`
   - `false-submission`
   - `materially-incomplete`
   - `procedural-failure`

2. `short reason`

3. `evidence url` または `evidence hash`

4. `requested outcome`
   - `refund`
   - `integrity penalty review`
   - `cooldown review`

長文の作文や法務的説明は要求しない。  
重要なのは、**第三者が後から最低限の事実関係を追えること**です。

## 6. report の効果

Phase1 の requester protection は、いきなり重い制裁を行う制度にはしない。

### 可能な結果

1. `Refund`
   - escrow の返還

2. `No reward`
   - worker への支払いなし

3. `Integrity penalty review`
   - 明白な虚偽や悪質行為がある場合のみ候補

4. `Cooldown review`
   - 繰り返し悪質なら新規 agreement 制限候補

### Phase1 でやらないこと

- 自動重罰
- 一発 BAN
- 曖昧な quality score 制裁
- 能力不足そのものへの直接ペナルティ

## 7. 手続の位置づけ

requester protection は、`reject` で足りない場合の例外フローとして扱う。

基本の流れは次です。

1. worker submits  
2. requester reviews  
3. requester は次から選ぶ  
   - `approve`
   - `reject`
   - `misconduct report`
4. report が出た場合だけ第三者 review に入る  
5. resolver / finalizer が最終判断する

つまり、requester 側の保護は dispute engine を共有してもよいが、  
意味としては **worker 救済の issue** と **requester 保護の misconduct report** を分けて考える。

## 8. worker issue との非対称性

Phase1 では、これを「完全対称の dispute 制度」にはしない。

理由は次です。

- requester は資金や発注権を持ちやすく、制度を圧力手段として使いやすい
- worker 側は、未払いリスクに対する保護を強く必要とする
- 両者に同じ強さの dispute weapon を渡すと、強い側の圧迫装置になりやすい

したがって、Phase1 の方針は次です。

- `worker issue` は強い救済手段として残す
- `requester report` は、証拠可能な misconduct に限定して持たせる

これは「片方だけ優遇」ではなく、**立場の違いに応じた非対称設計**です。

## 9. Phase1 の安全条件

requester protection を入れるなら、最低限次を守るべきです。

1. report 類型を限定する  
2. 主観評価を排除する  
3. short reason + evidence を必須にする  
4. resolver / finalizer が最終判断する  
5. 自動重罰にしない  

## 10. まとめ

Phase1 の requester protection は、次の一文で表せます。

> **依頼者にも異議申立ての入口は必要だが、それは「worker を気に入らない時の攻撃手段」ではなく、no-show・虚偽・重大な未達を報告する限定的な misconduct report として扱う。**

この設計により、FairSoil は次の2つを同時に守りやすくなる。

1. worker 側が一方的に踏み倒されないこと  
2. requester 側も明白な不履行や虚偽提出から守られること

Phase1 では、この最小限の双方向保護で十分です。

# FairSoil 裁定制度で避けたいアンチパターン（JA）

このメモは、FairSoil の dispute / adjudication 設計で「何を実装したいか」ではなく、**何を制度上の失敗として避けたいか**を整理するための短い設計ノートです。  
目的は、単に正しい結論を出すことではなく、**忖度・引き延ばし・疲弊戦術に偏らない裁定手続**を保つことです。

## 基本方針

FairSoil が避けたいのは、現実の裁判や行政手続でしばしば問題になる次の構造です。

- 当事者よりも、制度上の上位者に都合の良い判断が合理的になる
- 正しい側でも、争うだけで時間・費用・生活機会を失う
- 証拠そのものより、作文能力、組織内政治、資産量が勝敗を左右する

したがって FairSoil は、理想的な「完全自動の正義」を目指すのではなく、まずは**争いが長引くだけで弱い側が潰れない最小手続**を優先する。

## 避けたいアンチパターン

### 1. 当事者兼審判

- dispute の当事者と、裁定者や最終確定者が実質同じ立場にある状態
- 運営が利害関係を持つ案件を、そのまま自分で裁く状態

これは FairSoil における最も基本的な NG です。  
Phase1 では resolver / finalizer を分けるが、それでも**当事者と裁定者の分離**を最優先に扱う。

### 2. 人事・権限の一極集中

- resolver / finalizer を誰にするかを、単一の強い管理者が自由に決め続けられる状態
- 裁定者が「上に逆らうと外される」と感じる構造

これは形式上の中立を壊します。  
FairSoil では、将来的に role の差し替え権限自体も timelock / governance / 外部制度へ段階移行する前提を維持する。

### 3. 理由なき裁定

- 結論だけ出て、何の証拠・時系列・手続を見たのか残らない状態
- 後から第三者が検証できない裁定

FairSoil の裁定は、**結果だけでなく、その判断理由を追跡可能にすること**が必要です。  
最低限、証拠参照・時系列・簡潔な理由は audit trail に残す。

### 4. 引き延ばしが得になる制度

- 期限が曖昧で、放置や遅延そのものが交渉力になる状態
- 争いを長引かせた側が実質有利になる構造

現実の制度でよく起きる「勝っても大損」の一因はここにあります。  
FairSoil では、response / proposal / finalize に期限を置き、**時間を武器にする戦術**を減らす。

### 5. 争うだけで大損する制度

- dispute のための負担が重すぎて、正しい側でも泣き寝入りした方が合理的になる状態
- 小額案件でも裁定コストが過大な状態

FairSoil は「裁定の正しさ」だけでなく、**争うコストの上限**を意識する必要がある。  
小額案件は簡易処理、高額案件のみ外部裁定など、価値に応じて重さを変える。

### 6. 作文能力や制度知識が強すぎる側だけ有利

- 長文、法務っぽい書き方、複雑な手続に慣れた側が有利になる状態
- 証拠より“うまく書ける人”が勝ちやすい設計

FairSoil は、法廷級の文章競争を再現しない。  
最低限必要なのは、**短い理由 + evidence URL / hash + 時系列**であり、過剰な作文を必須条件にしない。

### 7. 資産量がそのまま正義になる制度

- deposit や担保の重さが、実質的に低資産者の発言権を奪う状態
- wallet size が裁定結果に影響する状態

資産差は spam 抑止の条件に使えても、裁定結果の中身には直接使わない。  
判断の主軸は `evidence / timeline / procedure / malicious behavior history` とする。

### 8. 恣意が見えないブラックボックス

- なぜその結論になったか分からない
- 同種案件で判断基準が揺れても検出できない
- 「偉い人に都合がよい判断」が埋もれる状態

FairSoil は、裁定の完全自動化よりも先に、**恣意が見えやすいこと**を重視する。  
公開ログ、role 表示、理由メモはこのために必要です。

### 9. 第三者性の名ばかり化

- ランダム選出や陪審と称していても、実際は同じ人・同じ派閥・運営近辺に偏る状態
- 母集団が小さいのに、形式だけ中立を装う状態

「第三者」というラベルだけでは不十分です。  
候補母集団が小さい段階では、ランダム化よりも**利害関係の除外とログ公開**の方が実効性を持つことがあります。

### 10. 例外処理が多すぎて何でも裁量で決まる制度

- ルールより個別判断が多く、結局“人を見る”制度になる状態
- 透明な手続より運営判断が優先される状態

FairSoil は、複雑な正義理論を直接コード化するより、**少数の明確な手続ルール**を優先する。  
複雑さはしばしば裁量の温床になる。

### 11. 再審や上書きが無制限

- 一度決まった案件を何度も蒸し返せる状態
- 負けた側が相手を疲弊させ続けられる構造

FairSoil では、再審や異議申立ての回数・条件を制限し、**争いを終わらせる出口**を明確に保つ。

### 12. 高額案件と低額案件を同じ重さで扱う

- すべてを本格裁定に回して制度疲労を起こす状態
- 低額案件でも重すぎる dispute を要求する状態

FairSoil は「何でも裁判化する制度」を目指さない。  
小額案件は軽く終える。重い裁定は高額・高リスク案件に限る。

## Phase1 の最小原則

Phase1 の dispute は、次を守れば十分です。

- 裁定結果は `wallet size` ではなく `evidence / timeline / procedure` を主軸に判断する
- role を分ける（resolver / finalizer）
- 理由とログを残す
- 期限を置く
- 小額案件では争いのコストを上げすぎない

Phase1 は「完全な独立裁判制度」ではなく、**弱い側が一方的に踏み倒されにくい最小防衛線**として位置づける。

## Phase2 以降で扱うべきこと

以下は重要だが、Phase1 の必須条件ではない。

- external arbiter / jury routing
- elected resolver
- resolver / finalizer 選出制度
- timelock / governance への role 移行
- high-value dispute routing

これらは、裁定の第三者性を強めるための段階的拡張として扱う。

## まとめ

FairSoil の dispute 設計で最も重要なのは、**理想的な正義を完全自動化すること**ではない。  
先に必要なのは、次の3つです。

1. 当事者と裁定者が混ざりすぎないこと  
2. 争いが長引くだけで弱い側が潰れないこと  
3. なぜその結論になったかを後から見返せること

この3点が守れないなら、複雑な陪審や外部裁定を足しても、制度は見た目だけ立派で中身の弱いものになりやすい。

# FairSoil Dispute Record 公開設計メモ（JA）

このメモは、FairSoil の dispute record を**どう公開可能に設計するか**を整理した短いノートです。  
目的は、FairSoil 自体を完全な最終裁判所にすることではなく、**揉め事を隠しにくくし、改ざんしにくくし、必要なら外部制度へ持ち出しやすくする記録基盤**として位置づけることです。

## 1. 基本方針

FairSoil の dispute は、密室で完結することを目指さない。  
同時に、常時フル公開の晒し制度にも寄せない。

重視するのは次の3点です。

1. **検証可能であること**  
2. **持ち出し可能であること**  
3. **時系列が追えること**

つまり、FairSoil の役割は「最終的にすべてを裁き切ること」よりも、  
**後から第三者が見ても、何が起きたかを追跡できる記録を残すこと**にあります。

## 2. なぜ公開設計が必要か

現実の dispute や裁判で問題になりやすいのは、しばしば次です。

- 記録が散らばる
- 誰が何を言ったか追いにくい
- 後から都合よく編集される
- 相手側だけが広報・組織力・メディア接続を持つ

FairSoil が dispute record を整理しやすくすることで、  
一方だけが情報を握りつぶしたり、片側だけが都合のよい narrative を広げたりする状態を少し弱められる。

## 3. 最低限残すべき記録

各 dispute では、最低限次を残すべきです。

### 3-1. timestamp

- いつ何が起きたか

### 3-2. actor

- 誰が操作したか

### 3-3. role

- requester
- worker
- resolver
- finalizer
- operator

### 3-4. action

- issue
- accept
- challenge / misconduct report
- proposal
- finalize

### 3-5. short reason

- 長文ではなく短い説明

### 3-6. evidence reference

- URL
- hash
- packet

### 3-7. outcome

- payout
- refund
- penalty candidate
- finalized status

## 4. 公開レベルは段階化する

FairSoil の dispute record は、すべてを即時フル公開するのではなく、  
**公開レベルを段階化**するのが望ましい。

### Level 1: 当事者向け詳細

- 当事者と裁定者だけが見える詳細記録
- 生の evidence URL や未整理メモを含んでもよい

### Level 2: 公開してよい要約版

- 公開可能な short summary
- 不要な個人情報やセンシティブ情報は落とす
- 第三者レビューやコミュニティ説明に使う

### Level 3: export packet

- 現実の裁判
- 弁護士相談
- 外部裁定
- 監査

にそのまま持ち出せる dispute packet

## 5. SNS 向けに重要なのは「煽り」ではなく「整理」

FairSoil が支援すべきなのは、炎上を起こすことではない。  
重要なのは、**感情的な投稿よりも、事実関係を整理して出せること**です。

SNS 向けに必要なのは次です。

- 何が起きたか
- いつ起きたか
- 何を証拠として出したか
- 相手がどう応答したか
- どの role がどう判断したか

つまり、`thread 化しやすい時系列メモ` と `証拠一覧` を出しやすくする方が価値が高い。

## 6. FairSoil 内で必要な UI

公開設計のため、FairSoil 内では次の UI を優先する。

### 6-1. timeline

- plain order で追えること

### 6-2. role 表示

- 誰がどの立場で動いたかが分かること

### 6-3. evidence preview

- URL / title / hash / summary を確認できること

### 6-4. export

- CSV
- markdown
- packet

などへ持ち出せること

### 6-5. public-safe summary

- そのまま共有しやすい短い版を作れること

## 7. 期待できる効果

この設計により、少なくとも次が期待できます。

- 一方的に情報を握りつぶしにくい
- 後から「言った / 言わない」を減らせる
- 第三者裁定の質を少し上げられる
- 現実裁判や外部相談へつなぎやすい
- 相手側だけが広報力を持つ状況を少し崩せる

## 8. 注意点

公開しやすさは重要だが、次を避ける必要があります。

- 晒し文化になること
- 個人情報をむやみに露出すること
- 公開圧力で弱い側がさらに疲弊すること
- SNS の炎上が裁定そのものを置き換えてしまうこと

FairSoil の dispute record は、**攻撃の拡声器**ではなく、  
**検証しやすい記録装置**として設計するべきです。

## 9. まとめ

FairSoil の dispute 設計は、

> **正義を完全自動判定する仕組み**

よりも、

> **証拠と時系列を整理し、隠しにくくし、必要なら外へ持ち出せる仕組み**

として考える方が強い。

Phase1 では、次を守れば十分です。

1. 誰が何をしたかが追える  
2. evidence reference が残る  
3. role と outcome が記録される  
4. 必要なら外へ持ち出せる

この4点があるだけでも、FairSoil は「密室で握りつぶされないための手続インフラ」として十分な価値を持ちうる。

# FairSoil の対外説明で誤解されやすい点と、その言い換え（JA）

このメモは、FairSoil を外部へ説明する時に誤解されやすい点を整理し、  
**どう言い換えると制度の実態に近いか**をまとめた短いノートです。

FairSoil は、単なる減価通貨でも、完全自動裁判所でも、特定の政治ラベルを押し出す運動体でもない。  
現時点では、**最低限の交渉力を支える通貨設計**と、**踏み倒しや情報の握りつぶしを減らす軽量な手続**を組み合わせた実験的な基盤です。

## 1. 「FairSoil は裁判を置き換える仕組みなのか」

### 誤解されやすい言い方

- 「FairSoil の中で全部裁ける」
- 「オンチェーンで正義を自動判定する」

### 現時点で近い言い換え

- FairSoil は、**揉め事を軽く処理する一次レイヤー**である
- 重い案件や本気で争う案件は、**現実の裁判・弁護士相談・外部裁定**へ持ち出せるようにする
- 役割は「最終裁判所」ではなく、**記録を整理し、外へ持ち出しやすくする手続基盤**である

### 補足

この点は `docs/dispute_record_publication_ja.md` と `docs/phase1_minimal_dispute_spec_ja.md` に対応している。

## 2. 「第三者裁定を入れれば公平になるのか」

### 誤解されやすい言い方

- 「第三者に任せれば中立になる」
- 「陪審やランダム選出にすれば問題は消える」

### 現時点で近い言い換え

- 第三者裁定は必要だが、**それだけで公平性は完成しない**
- 人は他人の dispute に強い関心を持たないし、回答忘れ、雑な判断、好き嫌い、身内びいきは起こりうる
- だから重要なのは、第三者を神格化することではなく、**期限、理由の記録、利害関係者の除外、公開可能な記録**を組み合わせること

### 補足

FairSoil が目指すのは、理想的な裁定者を前提にすることではなく、  
**雑な裁定や恣意が起きても後から見えやすい制度**に寄せることです。

## 3. 「Token A があるなら、それだけで生活できるのか」

### 誤解されやすい言い方

- 「FairSoil の UBI だけで今すぐ生活できる」
- 「100 Token A / 日 は最低生活費を保証している」

### 現時点で近い言い換え

- 現在の `Token A` は、**制度挙動確認用の初期値**であり、生活保障の完成値ではない
- まずは、ひどい条件を断っても即ゼロにならないようにする、つまり**交渉拒否権を少し回復する安全装置**として読むのが近い
- 現実の生活費に近づけるには、財源、物価、地域条件、供給網を踏まえた次段階の校正が必要

### 補足

この点は `docs/token_a_calibration_notes_ja.md` に整理している。

## 4. 「再分配や UBI の話だから、思想的に偏って見えないか」

### 誤解されやすい言い方

- 「FairSoil は特定の政治思想の実装だ」
- 「再分配 ideology を押し付ける仕組みだ」

### 現時点で近い言い換え

- FairSoil は、**弱い立場でも一方的に踏み倒されにくい最低限の安全装置**として説明した方が近い
- 目的は、価値観の強制よりも、**誠実な取引を壊しにくいインフラ**を作ることにある
- UBI も dispute も、理想社会の完成品ではなく、**極端な搾取や黙らせを少し減らすための実務的ガードレール**として位置づける

### 補足

対外説明では、政治ラベルの訴求より、

- 最低限の交渉力を維持する
- 情報を握りつぶされにくくする
- 踏み倒しをしにくくする

という実務的な効用を前に出す方が誤解が少ない。

## 5. 「FairSoil は雇う / 雇われる関係しか扱えないのか」

### 誤解されやすい言い方

- 「Covenant があるなら、送金も全部それで十分」
- 「FairSoil は work agreement だけの制度だ」

### 現時点で近い言い換え

- Phase1 の中心は `work agreement` だが、長期的にはそれだけでは足りない
- 将来的には、少なくとも次のような**通常の受け渡し**も別の型として必要になる
  - simple transfer
  - gift
  - reimbursement / shared expense
  - 小さな物品売買や近所間の普通の送金

### 補足

work agreement は「揉めやすい仕事取引」を扱う型であって、  
日常のすべての価値移転を 1 つの重いフローに閉じ込めるべきではない。

## 6. 「公開しやすくするなら、晒しや炎上を促す制度ではないか」

### 誤解されやすい言い方

- 「全部公開して世論裁判にすればいい」
- 「SNS で燃やせば抑止になる」

### 現時点で近い言い換え

- FairSoil が目指すのは、**炎上のための拡声器**ではなく、**検証しやすく持ち出しやすい記録装置**
- 公開は即時フル公開ではなく、`当事者向け詳細 / 公開可能な要約 / export packet` のように段階化する
- 重要なのは、感情的な告発より、**何が起きたかを後から追えること**

### 補足

この点は `docs/dispute_record_publication_ja.md` に詳しい。

## 7. 現時点の短い説明テンプレート

外向けには、現時点の FairSoil を次のように説明するのが比較的ズレにくい。

> FairSoil は、減価する生活通貨と、escrow 付きの仕事取引、そして軽い dispute record を組み合わせて、  
> 一方的な踏み倒しや情報の握りつぶしを減らそうとする基盤である。  
> それ自体が最終裁判所ではなく、必要なら現実の制度へ持ち出しやすい記録と手続を整えることを重視している。

## 8. まとめ

FairSoil の対外説明で大切なのは、

- 理想を大きく言いすぎない
- 何がまだ未完成かを隠さない
- それでも何を改善しようとしているかを具体的に言う

ことです。

現時点では、次の 4 つを押さえるだけでかなり伝わりやすくなります。

1. FairSoil は裁判の代替ではなく、軽い保護レイヤーと記録基盤である  
2. Token A は生活完成額ではなく、交渉拒否権を少し回復する初期設計である  
3. FairSoil は政治ラベルより、踏み倒しや握りつぶしを減らす実務的な安全装置として説明する方が近い  
4. work agreement だけで全てを扱うのではなく、将来的には普通の送金や立替精算の型も必要になる
